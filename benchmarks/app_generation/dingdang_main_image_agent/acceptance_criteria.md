# Dingdang Benchmark 验收标准

## 状态

本文档定义 `dingdang_main_image_agent` benchmark 的业务验收标准，用于评估 PRD 生成应用的结果。它不是功能实现计划，也不要求本阶段新增代码。

## AC-D001 PRD 理解完整

生成结果必须识别 Dingdang 的目标用户、核心场景和价值主张：为电商运营者、店铺老板和设计师生成一套可投放、可 AB 测试的 8 张主图。

验证信号：

- 标准化 PRD 包含目标用户和核心场景。
- 应用界面或输出能表达“8 张主图”“可投放”“可 AB 测试”“可局部迭代”。

## AC-D002 四阶段流程可见

生成应用必须体现四阶段流程：需求诊断、创意方案、策略落地、Prompt 生成与执行。

验证信号：

- 用户能看到当前阶段。
- 每个阶段有输入、输出或状态摘要。
- 阶段顺序符合 PRD。

## AC-D003 Stage 1 有任务类型确认阻断

需求诊断后必须要求用户确认任务类型，不能自动进入创意方案。

验证信号：

- 用户在未确认前不能进入 Stage 2。
- 任务类型至少区分“完整 8 张主图”和“8 张首图测点击率”。

## AC-D004 Stage 2 方案单选且禁止混搭

创意方案阶段必须输出 2 到 3 套方案，并要求用户选择一套。应用不得允许把不同方案的单张图混搭成最终方案。

验证信号：

- 方案选择是单选。
- UI 或业务规则明确禁止混搭。
- 已选方案成为后续策略落地的来源。

## AC-D005 平台策略差异

应用必须支持平台策略差异，至少覆盖天猫、淘宝、抖音、拼多多。

验证信号：

- 用户能选择或查看平台。
- 不同平台有不同策略提示或设计禁忌。

## AC-D006 8 张图分工与一致性约束

应用必须表达 8 张主图的分工、统一视觉基因和差异变量。

验证信号：

- 有 8 张图的规划结构。
- 能看到每张图的角色或目标。
- 能看到共享约束，例如色温、光影、产品角度或统一基因。

## AC-D007 Prompt 分层

应用必须能表达每张图的分层 Prompt 或可执行施工图。

验证信号：

- 至少区分产品锚定层、场景/氛围层、文案/卖点层或等价分层。
- 每张图可查看 prompt 细节。

## AC-D008 局部迭代

应用必须支持“第 X 张第 Y 层”的局部迭代表达。

验证信号：

- 用户能定位到某张图和某个层级。
- 修改建议不会要求整套返工。

## AC-D009 本地原型可运行

生成应用必须可本地预览，并记录清晰启动方式。

验证信号：

- 有 README 或 preview instructions。
- 语法检查或测试通过；失败时记录清晰 blocker。

## AC-D010 产品图与参考图上传

生成应用必须支持用户上传产品图和参考图。产品图是进入真实出图链路的必填输入，参考图是可选输入但必须在 UI 和执行请求中被支持。

验证信号：

- UI 至少包含产品图上传控件和参考图上传控件。
- 产品图上传后可被转换为图片 data URL 或等价本地可传输格式。
- 参考图上传后可用于参考图类型判断、prompt 或图片 provider 请求。
- 未上传产品图时，真实出图按钮必须给出清晰提示，不能静默失败或继续生成。

## AC-D011 显式图片 Provider 与出图

生成应用必须支持显式配置的图片 provider，至少支持 OpenAI 或 OpenRouter 之一，推荐同时支持两者。API key 只能在 Node 服务端读取，前端不得持久化或展示 secret。

验证信号：

- 本地服务提供图片生成 API，例如 `/api/images/generate`。
- `.env.example` 只包含占位 key 和 provider 配置示例。
- UI 能显示 provider 是否已配置。
- 用户能单张生成图片，也能批量生成 8 张图片。
- provider 未配置、模型不支持参考图或请求超时时，界面显示清晰错误。

## AC-D012 Prompt 与图片下载

生成应用必须支持下载 Prompt 和生成图片。若 provider 返回远程 URL，下载链路必须通过服务端代理或 blob 化处理，不能依赖可能被跨域限制忽略的裸 `<a download>`。

验证信号：

- 每张图可下载 Prompt。
- 每张已生成图片可下载。
- provider 返回 base64 data URL 和远程 URL 两种情况都有明确处理策略。

## AC-D013 安全与边界

生成结果不得保存真实 secret，不得生成数据库或隐藏部署流程。若接入图片 provider，必须显式声明配置和限制。

验证信号：

- 不包含 `.env` 或真实 key。
- 不生成数据库连接、迁移或持久化服务。
- 外部 API 调用不是隐藏行为。
