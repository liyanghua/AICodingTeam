App Generation 三条编译链路 V2：接口契约版

这张图把系统拆成三条可独立建设、又必须通过接口闭合的链路：业务文档编译链路、运行时/节点执行流编译链路、前端 UI 与 Agent 交互链路。V2 重点把“模块依赖”改成“接口契约依赖”：前端不直接依赖索引查询，Agent 不直接理解整套状态机，而是消费稳定的引用证据契约和交互动作契约。

总览图：三条编译链路如何协同

这张图可以作为 V6 后续的细化页：它把“业务蓝图编译器”“图执行引擎”“生成工作台”拆成三个可分阶段建设的编译器。V2 的关键变化是把依赖表达为契约：中间产物协议依赖引用与证据契约，右侧 Agent 协议依赖运行状态、人机介入点和允许动作。

三条链路的职责边界

1. 业务文档编译链路

负责把业务资料变成可靠的业务语义资产，重点不是“总结文档”，而是形成可查询、可引用、可验证的业务上下文。

  • 解析：策略文档、行业资料、企业数据说明、私有模型说明。
  • 索引：段落、术语、证据、口径、来源。
  • 查询：wiki 化问答、证据引用、业务假设回溯。
  • 输出：PRD++、ScenarioSpec、DataSpec、KnowledgeSpec、Reference Contract、Evidence Contract。

2. 运行时 / 节点执行流编译链路

负责把业务运行资产变成可执行、可观测、可重跑的节点图，重点是执行结构和状态边界。

  • 节点类型：文本、图片、计算、表单、工具、预览、验证、人工审核。
  • 节点契约：输入、输出、中间产物、证据、风险、依赖关系。
  • 执行规则：DAG、状态机、门禁、失败分类、人工介入点。
  • 输出：BusinessStep、NodeSpec、RunState、HumanGate、allowed_actions、ArtifactContract。

3. 前端 UI 与 Agent 交互链路

负责把执行图和中间产物变成用户可理解、可操作的工作台，重点是组件渲染和 human-in-the-loop。

  • 页面结构：左侧任务、中间工作区、右侧 Agent。
  • 组件匹配：文本节点、图片节点、可计算节点、表单节点、预览节点、证据节点。
  • 交互协议：CanvasSelectionContext、AgentAction、repair request、rerun request。
  • 契约消费：中间产物消费 Reference/Evidence，右侧 Agent 消费 RunState/HumanGate/allowed_actions。
  • 介入方式：解释、确认、编辑、重跑、修复、提升为生成规则。

V2 关键修正：从模块依赖改为接口契约依赖

中间产物协议不直接依赖索引查询

索引和查询是底层能力。中间产物协议真正消费的是引用体系：source_refs、evidence_refs、knowledge_refs、artifact_refs。这样前端和 Agent 不需要知道索引服务如何实现，只需要能追溯来源和证据。

右侧 Agent 协议依赖交互边界

右侧 Agent 需要知道当前步骤能不能解释、能不能编辑、是否允许重跑、是否需要人工确认、是否能委托 Code Agent。这些来自 RunState、HumanGate、allowed_actions 和 artifact permission。

前端工作台消费两个稳定接口

前端渲染对象时消费 Reference/Evidence Contract，执行动作时消费 Interaction/Action Contract。这样三栏 UI 可以迭代，底层索引、图执行引擎和 Agent Provider 也可以独立演进。

判别信号：这三条链路是否真的跑通

业务资料可查询 用户问“这个判断来自哪里”，系统能回到原始文档、数据口径和证据。
节点图可执行 每个节点都有输入、输出、中间产物、依赖、状态和失败处理。
组件可匹配 文本、图片、计算、预览、证据等节点能被合适组件展示，而不是统一渲染成日志。
Agent 可介入 用户在右侧提出修改，系统能转成确认动作、节点重跑或 Code Agent 修复。

MVP 可以先人工整理业务资料和确认业务口径，但三条链路的接口必须产品化：文档资产要输出引用和证据契约，节点流要输出运行和动作契约，前端反馈要能回写为证据和规则候选。
