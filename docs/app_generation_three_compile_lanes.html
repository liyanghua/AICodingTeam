App Generation 三条编译链路

这张图把系统拆成三条可独立建设、又必须通过接口闭合的链路：业务文档编译链路、运行时/节点执行流编译链路、前端 UI 与 Agent 交互链路。三条线并不是三套系统——它们共享同一份节点契约层（single source of truth）：文档给出业务真相，节点流给出执行结构，工作台给出可理解、可介入、可修复的操作现场。

总览图：三条编译链路如何协同

与旧版相比的四处结构修正：① 抽出贯穿三条线的共享节点契约层，节点类型只定义一次，Lane3 渲染组件改为“按契约消费”而非自列清单；② Lane2 拆成编译子段（定义结构）与执行子段（运行实例）两个生命周期；③ 把 human-in-the-loop 标成声明（Lane2 HumanGate）↔ 实现（Lane3 交互）的对应关系；④ 运行证据收敛到单一落点（Lane2 执行子段），回写线目标层各自明确。

三条链路的职责边界

1. 业务文档编译链路

把业务资料变成可靠的业务语义资产，重点不是“总结文档”，而是形成可查询、可引用、可验证的业务上下文。

  • 解析：策略文档、行业资料、企业数据说明、私有模型说明。
  • 索引：段落、术语、证据、口径、来源。
  • 查询：wiki 化问答、证据引用、业务假设回溯。
  • 输出：PRD++、ScenarioSpec、DataSpec、KnowledgeSpec、ToolSpec、EvalSpec、EvidenceSpec（证据定义）。

2. 运行时 / 节点执行流编译链路

分两个生命周期：编译子段定义静态结构，执行子段负责动态运行实例，两段边界不能混。

  • 编译子段：节点类型（引用共享契约）、I/O 与中间产物、依赖 DAG。
  • 执行子段：状态机、门禁、失败分类、人工介入点声明（HumanGate 哪里/何时）。
  • 运行证据的单一落点：run artifacts 由执行子段产出，不与文档层证据定义混同。
  • 输出：BusinessStep、NodeSpec、ArtifactContract、RunState、VerificationGate。

3. 前端 UI 与 Agent 交互链路

把执行图和中间产物变成可理解、可操作的工作台，重点是按契约渲染与 human-in-the-loop 的实现。

  • 页面结构：左侧任务、中间工作区、右侧 Agent。
  • 组件匹配：读共享契约按节点类型选渲染器，不再自列类型清单。
  • 介入实现：把 Lane2 声明的 HumanGate 落成具体交互时机与操作方式。
  • 协议：CanvasSelectionContext、AgentAction、repair request、rerun request。

共享节点契约层是三条线的公共依赖，不属于任何单独一条：Lane1 供给业务字段 schema、Lane2 编译子段定义类型与契约、Lane3 按契约消费渲染。它一旦漂移，三条线会同时失效——因此节点类型与契约字段必须只有一处定义。

判别信号：这三条链路是否真的跑通

业务资料可查询 用户问“这个判断来自哪里”，系统能回到原始文档、数据口径和证据。
契约只定义一次 新增一种节点类型，只改共享契约层，Lane2 执行与 Lane3 渲染自动对齐，无需两处同步。
编译与执行分离 节点结构在编译子段确定，运行状态与人工门禁在执行子段实例化，两者互不越界。
HITL 声明即实现 Lane2 声明某节点需人工介入，Lane3 就能自动给出对应交互入口，证据统一回写。

MVP 可以先人工整理业务资料和确认业务口径，但三条链路的接口必须产品化：文档资产要能被节点消费，节点契约要能被前端渲染，前端反馈要能回写为运行证据（单一落点）和知识索引候选。
