# 数据分析节点生成与交互规范

## 状态

本文档定义 `app_generation` 中“数据分析节点”的产品与交互规范。它回答一个问题：业务策略文档里按流程写出的“目的、数据来源、执行动作、表格字段、分析结论”，在生成应用里应该如何变成一个可运行、可审核、可追溯的数据分析节点。

本文档是规范文档，不修改 `tasks/current/*`，不手改历史 `runs/.../generated_apps/...`，不默认启用真实数仓 live probe。中间工作区仍是事实源，右侧 Agent 仍是协作层。

## 背景

市场洞察类业务策略文档通常不是简单的表单说明，而是一个分析流程。例如“流程2：行业大盘与热销商品分析”同时描述：

- 为什么分析：目的。
- 从哪里拿数据：数据来源。
- 怎么做：执行动作。
- 产出什么表：表格字段。
- 怎么形成判断：分析结论。

如果生成应用只把这些内容压成一张“字段覆盖表”，用户会看不清节点到底在做什么：输入为什么需要这些数据、字段匹配服务为什么选这些 API、执行动作如何使用字段、分析结论又从哪些证据得出。

因此，数据分析节点需要一个稳定的运行范式，把业务文档的结构映射为应用里的可操作区块。

## 核心定义

### 数据分析节点

数据分析节点是指：节点目标需要读取或匹配外部/上游数据，按照业务文档中的执行动作加工数据，并同时产生结构化数据表和分析结论的节点。

典型节点：

- 行业大盘与热销商品分析。
- 价格带/人群/场景趋势分析。
- 竞品卖点与评价反馈分析。
- 内容表现与转化归因分析。

非目标节点：

- 纯手填项目定义表。
- 纯展示节点。
- 不涉及字段映射、数据证据或分析结论的普通计算节点。

### 节点运行范式

数据分析节点的运行由五个业务对象组成：

| 业务文档片段 | 节点对象 | 用途 |
| --- | --- | --- |
| 目的 | 节点目标 | 说明当前节点要解决什么业务问题，以及用户为什么要做这一步 |
| 数据来源 | 输入准备 | 决定需要哪些数据、上游产物、业务参数和 API 能力 |
| 执行动作 | 运行过程 | 决定用户和系统按什么步骤完成节点 |
| 表格字段 | 数据表产物 | 决定结构化输出字段、字段覆盖和 API 字段映射 |
| 分析结论 | 结论产物 | 决定如何基于数据表和证据形成业务判断 |

输入由 `数据来源 + 表格字段 + 上游产物` 共同决定。运行过程由 `执行动作` 决定。输出分为两类：数据表产物和分析结论产物。

### P0 / P1 边界

P0 已覆盖“节点视图和字段覆盖”：用户能在中间工作区看清目的、输入、字段覆盖、执行动作和输出要求，并能把业务字段映射到真实 API 返回字段。

P1 进入“节点执行和草稿产物”：用户点击“运行当前节点”后，系统按执行动作读取上游产物参数，显式开启 live probe 时调用数仓 API 取数，生成数据表草稿；派生字段由 PI/LLM 基于数据证据生成草稿；分析结论基于数据表和业务文档要求生成草稿。P1 仍不自动确认字段映射、派生字段或分析结论。

## 业务文档到节点模型的映射

### 目的 -> 节点目标

`目的` 应转译为中间工作区顶部的“节点目标”：

- 业务问题：例如“判断谁涨得快、谁稳定、谁赚钱”。
- 分析价值：例如“通过行业数据判断流行趋势和机会点”。
- 成功标准：例如“最终能识别热销商品结构、价格带和爆款原因”。
- 来源追踪：指向原始业务文档流程段落。

节点目标不是 Agent 总结文案，而是运行节点前用户必须理解的业务意图。

### 数据来源 -> 输入准备

`数据来源` 应转译为“输入准备”：

- 上游产物：例如第一步《市场洞察项目定义表》里的类目、周期、产品线。
- 外部数据来源：例如生意参谋、类目排行、商品排行、关键词或评价数据。
- 必填业务参数：例如类目 ID、分析周期、价格带、平台、榜单类型。
- 数据能力入口：用于触发 API 文档匹配或数仓能力匹配。
- 缺口状态：哪些参数缺失，哪些数据源未配置，哪些字段需要人工补充。

输入准备区只解决“数据从哪里来、够不够用”的问题，不得直接生成业务结论。

### 执行动作 -> 运行过程

`执行动作` 应转译为“运行过程”清单：

- 每个动作是一个可检查步骤。
- 每个步骤声明依赖输入。
- 每个步骤声明产生的中间结果。
- 每个步骤声明是否需要人工确认。

示例：

1. 根据项目定义表确定类目和周期。
2. 获取类目排行和商品排行候选 API。
3. 将表格字段匹配到 API 返回字段。
4. 生成行业前 300 商品分析表草稿。
5. 基于表格数据形成趋势、机会和风险结论草稿。

运行过程不应该藏在右侧 Agent 对话里。右侧 Agent 可以解释或纠错，但主流程在中间工作区。

点击“运行当前节点”时，运行过程应按执行动作清单推进，而不是只返回 `waiting_upload`。对于数据分析节点，P1 的运行语义是：

1. 从上游产物读取业务参数，例如类目、周期、产品线、价格带。
2. 检查字段覆盖方案，缺失时先触发字段覆盖生成。
3. 将业务语义参数绑定到每个候选 API 的真实请求参数，例如 `category` -> `cat_id`、`period` -> `date_range`。
4. 将已覆盖字段按来源 API 分组，形成取数计划。
5. 在 `DBA_LIVE_PROBE=1` 且数仓助手可用时调用数仓 API。
6. 用 API 返回字段填充数据表草稿中的原生字段。
7. 将派生/人工字段交给右侧 Agent/PI 生成草稿建议。
8. 基于数据表草稿和分析结论要求生成结论草稿。

如果未开启 live probe、缺少真实数仓能力，或业务参数无法绑定到 API 必填请求参数，节点应展示“取数受阻/待配置”和可行动的缺口问题，不得生成假数据。

参数绑定必须在中间工作区可见。用户应该看到每个 API 的请求参数状态：

- `bound`：已从上游产物、节点输入或用户补充中绑定。
- `missing`：API 必填参数缺失，当前 API 不能调用。
- `manual_required`：系统能识别语义接近的参数，但需要用户确认口径。
- `unsupported`：P1 不支持的依赖型参数，例如需要先从另一个 API 取 `product_id` 列表。

日期参数需要展示转换依据，而不是只显示最终值。比如用户选择 `近30天`，执行日期为 `2026-07-09` 时，用户应能看到：

- `deal_date` / `biz_date` / `dt` 这类单日参数被绑定为 `2026-07-09`。
- `start_date` / `end_date` 被绑定为 `2026-06-10` / `2026-07-09`。
- `statist_date` / `month` 这类月度参数被绑定为 `2026-07`。
- `update_time`、`created_at`、`modified_time` 等审计时间不会自动绑定业务周期，避免把分析周期误当作数据更新时间。

P1 只支持从当前 known params 直接完成的独立 API 调用。若某个 API 依赖另一个 API 的返回值作为入参，应标记为 `dependency_required`，进入 P2 依赖取数编排，不在 P1 静默猜测。

### 表格字段 -> 数据表产物

`表格字段` 应转译为“数据表产物”：

- 字段名保持业务文档原始中文表达。
- 字段说明必须保留，例如“商品主图：看视觉表达”。
- 每个字段展示覆盖状态、来源 API、API 字段路径、置信度和人工备注。
- API 字段匹配由 `api_doc_matcher` 或后续数据能力服务提供，前端不实现独立匹配算法。
- 派生字段不能强行误配为 API 原生字段，应标记为派生/人工。

数据表产物是第一类输出。它可以是草稿、已覆盖、待确认或已确认，但只有人工确认后才能作为后续结论的事实输入。

P1 数据表产物分为两层：

- 字段覆盖层：字段来源 API、返回字段路径、置信度和人工备注。
- 数据行草稿层：真实/样例 API 返回值按字段覆盖方案投影后的行数据。

字段覆盖层必须精确到 `source_field_path` 粒度，而不是只说明“来自某个 API”。如果一个字段只有 `source_api_id`，但没有可投影的 `source_field_path`，该字段不能进入数据行投影，应显示为“字段路径缺失/待确认”。

数据行草稿层必须区分“字段已匹配”和“真实值已取到”。API 返回成功并不代表字段值齐全；每个字段都要显示取值状态，例如已取到、缺失、空值、API 未调用或 join 受阻。必填字段未取到值时，输出结果应是 `partial_data_table_ready` 或 blocked，而不是 `data_table_ready`。

派生字段在节点取数结束时保持待补充。只有用户从单元格加入右侧 Agent 对话并显式回填后，数据行草稿层才出现 PI/LLM 草稿值；该值必须保留证据字段和未确认状态。无证据时只能显示填充方案或待补充问题。

### 分析结论 -> 结论产物

`分析结论` 应转译为“结论产物”：

- 结论要求：业务文档要求回答什么问题。
- 证据依赖：结论依赖哪些数据字段、上游产物或样例行。
- 结论草稿：可由 Agent 或规则生成，但必须标记为草稿。
- 风险和缺口：数据不足、字段缺失、口径不一致时必须展示。
- 人工确认：结论必须经过用户确认后才能进入节点事实。

分析结论是第二类输出。它不是字段映射合同，也不是 API 返回 JSON；它是基于数据表和分析要求形成的业务判断。

P1 的分析结论由用户点击右侧分析问题后生成独立 `insight` 草稿，必须展示：

- 使用了哪些数据字段或样例行。
- 哪些判断来自派生字段。
- 哪些结论存在数据缺口或口径风险。
- 当前是否经过用户确认。

没有证据引用的结论不得进入节点事实源。

## 中间工作区交互

数据分析节点的中间工作区应按五段组织。

### 1. 节点目标

展示：

- 当前节点目的。
- 原始业务文档流程标题。
- 判断标准或验收意图。

用户动作：

- 查看该节点为什么存在。
- 确认当前节点是否符合业务目标。

### 2. 输入准备

展示：

- 数据来源。
- 上游产物和已识别参数。
- 缺失参数。
- 每个候选 API 的请求参数绑定状态。
- 数据能力/API 匹配入口。

用户动作：

- 检查类目、周期、产品线等参数。
- 补充或确认 API 请求参数口径。
- 补充缺失输入。
- 触发字段覆盖方案生成。

### 3. 字段覆盖

展示：

- 表格字段要求。
- API/数据能力字段覆盖结果。
- 候选 API 返回字段。
- 派生/人工字段。
- 置信度和人工备注。

用户动作：

- 一键生成字段覆盖方案。
- 手动选择 API 返回字段。
- 应用高置信建议。
- 对错配字段发起右侧 Agent 纠错。

### 4. 执行动作

展示：

- 从业务文档执行动作转译出的步骤清单。
- 每步依赖的输入。
- 每步产出的中间结果。
- 是否需要人工确认。

用户动作：

- 检查节点是否按业务 SOP 运行。
- 标记某步暂不能执行。
- 查看某步依赖哪些数据字段。
- 点击“运行当前节点”，按执行动作推进取数、字段投影、派生字段和结论草稿生成。
- 查看每个步骤的状态：待执行、执行中、已完成、受阻、降级。

### 5. 运行工具条与唯一数据表

数据分析节点不再设置独立“输出结果”“数据表产物”或“节点运行输出”区。中间工作区只展示一张 `data-table-workspace-v1` 可编辑表格：

- 工具条展示运行状态、真实行数、字段覆盖率和“运行当前节点”。
- 取数成功后按 TOP N 结果展示、分页和编辑；未取数时不生成假数据行。
- 单击单元格后，在表格下方显示固定操作条：商品身份、字段名、当前值、来源状态、人工输入、保存、加入 Agent 对话和恢复 API 原值。
- API 原值、人工覆盖、Agent 草稿和待补充状态必须可区分；人工或 Agent 覆盖不得抹除 API 原值与审计信息。
- 类目解析、API 请求、详情补全、逐字段取值状态和风险明细放入表格后的折叠“工程证据与来源追踪”，不再重复渲染数据表。
- 刷新页面后直接从服务端持久化工作区恢复表格，不依赖浏览器内存中的最后一次运行响应。

分析结论不在中间工作区重复展示。业务文档中的分析要求以右侧 Agent 顶部的文字链接呈现，结论草稿、证据绑定、人工确认和 stale 状态均在右侧连续对话中完成。

完成分页补齐和复核后，用户必须点击“确认当前表格并进入下一步”。系统以当前 workspace revision 的 `effective_rows` 固化 `data-table-confirmed-v1`，并把它作为下游节点的唯一表格输入。运行中的批量任务会阻止确认；未采纳的 Agent 建议在确认时标记为 `ignored_on_finalize` 并保留审计。确认后再次编辑表格，确认状态自动变为 `stale`，必须重新确认才能继续作为当前事实。

流程2运行前提供 TOP N 预设选择 `10 / 20 / 30 / 50`，默认 `20`。选择值同时约束主榜单返回行、商品详情 fan-out 数量和最终表格行数；分页固定每页10行。重新运行产生新的 execution，已有人工覆盖按稳定商品 ID 迁移，未进入新 N 范围的覆盖进入 orphaned 记录。

## 右侧 Agent 边界

右侧 Agent 是数据表与分析结论的统一协作入口，但不是节点执行器。它使用一条节点级连续对话，承担：

- 单元格补齐与纠错：用户把任意单元格加入对话，Agent 给出可显式回填的建议。
- 缺口解释：解释为什么某字段缺失、为什么某 API 不足。
- 分析问答：用户点击业务文档分析问题后，Agent 基于当前表格 revision、最多 100 行有效数据和确定性证据摘要开始多轮分析。有效数据必须使用工作区覆盖后的值，并为每行附加服务端生成的稳定 `row_id`；模型不得用数组下标替代商品身份或证据引用。
- 结论协作：在聊天回复中形成结论草稿、证据绑定、风险和待确认问题。

“加入 Agent 对话”只附加经过服务端校验的单元格上下文并聚焦输入框，不立即调用模型。分析问题文字链接点击后立即发起一次分析请求。单元格提案只有点击“回填此单元格”后才写入表格；结论提案只有人工保存、绑定证据并确认后才成为当前事实。

发送给 Agent 的表格上下文不是原始 API 全量响应。单元格补齐只发送当前选中商品的有效行和对应 `row_meta`；自由问答与分析问题由服务端从 `data-table-workspace-v1` 计算当前有效行，合并人工覆盖值，最多截取 100 行，并注入稳定 `row_id`。超过上限时使用字段分布、缺失率和 TOP/Bottom 摘要并标记 `evidence_truncated`。

右侧 Agent 不得：

- 自动确认字段映射。
- 自动确认分析结论。
- 在用户点击回填或保存前直接写节点事实源。
- 默认 live probe。
- 把没有证据的数据判断写成业务结论。
- 绕过中间工作区直接触发真实取数。

## 状态与事实边界

数据分析节点至少需要区分以下状态：

- `input_ready`：输入参数和数据来源足够进入字段覆盖。
- `field_mapping_suggested`：字段覆盖方案已生成，待人工审核。
- `field_mapping_reviewed`：用户已审核字段覆盖，但未取真实数据。
- `data_table_ready`：数据表产物已生成或确认。
- `data_table_blocked`：字段覆盖已完成，但真实取数因配置、参数或 API 能力不足受阻。
- `insight_draft_ready`：分析结论草稿已生成。
- `insight_confirmed`：用户确认分析结论可作为节点事实。
- `blocked`：缺参数、缺数据能力、权限不足或外部工具不可用。

未确认字段映射不得成为数据表事实。未引用证据的分析结论不得成为节点事实。

## 验收标准

产品验收：

- 数据分析节点不再只显示字段覆盖表。
- 中间工作区能看到节点目标、输入准备、字段覆盖、执行动作、运行工具条和唯一数据表。
- 表格字段保持业务文档原始中文表达和字段说明。
- 页面不重复出现“输出结果”“数据表产物”或“节点运行输出”表格。
- 右侧 Agent 以一条服务端持久化对话处理单元格纠错、缺口、分析问答和结论确认，不承担真实取数。

流程2验收：

- 能展示“行业大盘与热销商品分析”的目的。
- 能展示数据来源。
- 能展示执行动作。
- 能展示 17 个业务表格字段。
- 右侧以可点击文字链接展示全部分析结论要求，不使用下拉框。
- 一键字段覆盖后，字段仍以中文业务字段展示。
- 派生字段进入派生/人工处理，不被强行误配。
- 显式开启 live probe 后，点击“运行当前节点”能从第一步产物读取类目、周期、产品线，并生成数据表草稿或明确取数受阻原因。
- 有数据表草稿时，单元格和分析结论只生成待人工应用的草稿，并显示证据字段与风险。
- 刷新或重启后，表格、Agent 对话、提案和结论状态均从服务端恢复。

安全验收：

- 不默认调用真实数仓；真实取数必须显式开启 `DBA_LIVE_PROBE=1`。
- 不展示 `.env`、API key、cookie、数据库连接串。
- 不把 Agent 草稿结论当作已确认事实。

## 流程3：爆款基因提炼

流程3只读取流程2人工确认后的 `data-table-confirmed-v1`，不读取未确认草稿，也不重新调用数仓。系统按每个稳定 `row_id` 生成九维画像：产品类型、材质、功能、风格、人群、场景、价格带、视觉表达和流量入口。

- 产品类型、材质、功能、风格和场景继承确认表事实；Agent 只能规范化标签，不能覆盖原值。
- 人群和流量入口只能生成有证据的 `pi_derived_unconfirmed` 草稿；视觉表达没有主图元素或视觉输入时保持为空。
- 只有客单价时可生成明确标记的样本相对价格带，不能冒充生意参谋官方价格带。
- 页面展示逐商品画像、九维频次、聚合基因组合、分类信号和固定执行监视器。

强度分级作用于聚合基因组合，单个商品只展示其所属组合标签。规则使用 `matched / not_matched / unavailable / insufficient_sample` 四态证据；缺失证据不得按0处理。`N < 50`时允许生成画像和观察候选，但不得输出强爆款、趋势爆款或差异机会的确定性结论。区间买家数和交易指数可以展示，但不能替代支付买家数占比和GMV占比命中阈值。

流程3运行结果必须人工确认后写入 `hot-product-gene-analysis-confirmed-v1`。流程2 revision 变化后，已确认流程3结果自动进入 `stale`，后续机会分析不得继续使用。
