# 能力运行时（capability-runtime）架构白皮书

> 为什么 App 只是能力运行时的一层外壳，能力为什么要先以裸服务上线、再封装为 Skill，以及 Code Agent 的能力为什么能外溢、外溢需要哪些前置条件。

配套架构图：`docs/skill-runtime-architecture-v5.html`

---

## 0. 一句话

App 是能力运行时（capability-runtime）的一层可替换外壳；业务能力先以带契约的裸服务上线、再封装为 Skill，生成期与运行期共享同一套运行实体。Code Agent 越强，越应成为**确定性资产的生产者**（编译 / 造件 / 蒸馏），而不是**确定性的替代者**（运行期实时算事实）。

---

## 1. 背景：原始 MVP

最初的链路是一条直线：

```text
PRD（静态）-> task-breakdown-specification -> 应用模板 + 技术栈 -> Code Agent 一把梭 -> Generated App
```

特征：

- PRD 是静态根输入。
- Code Agent 是**唯一**应用出口。
- 业务能力（数据、知识、工具、评估）散落在 PRD 文字和生成代码里，没有独立形态。

这套能跑通 demo，但有三个结构性问题。

---

## 2. 我们遇到的结构性问题

1. **不确定性堆在最脆弱的一环**。整个 app 由一个概率性组件一次生成，验收一旦失败，分不清是 PRD 错、能力投影错，还是 codegen 抽风，debug 边界塌掉；而且没有 fallback。
2. **能力不可复用**。能力被 PRD 的产品形态焊死。同样的商机洞察能力，换成 API 而不是 workbench，就要重做。
3. **沉淀方向错误**。想从生成出来的 app 代码逆向抽 skill，但代码 = 业务能力 + UI 脚手架 + 技术栈胶水 + 随机风格，从信息最脏的下游往回捞，永远 distill 不干净。

---

## 3. 核心设计判断

### 3.1 App = 壳(A) + 能力运行时(B)

把 App 切成两半，边界划在「界面 vs 业务」：

- **壳 (A)**：UI、页面、状态机。自包含、可漂移、可重生成、可离线预览。**不是事实源**。
- **能力运行时(B)**：业务逻辑、契约、知识、评估、证据。**运行实体共享、单一真相、可回放**。

判定规则：**会漂移、可丢弃、和形态绑定的 → 编进 app；要单一真相、要回放、要回写、跨 app 复用的 → 留服务。**

A/B 的本质区别：A 共享的是「定义」，B 共享的是「执行 + 状态 + 证据」。只有 B（运行实体共享）能支撑"运行 → 反馈 → 回写"的闭环；A（规格共享）会随每次生成漂移，反馈散落收不回来。

### 3.2 Code Agent 不是唯一载体：三路分流

按确定性把"应用"拆开，分别走不同载体：

| 路线 | 处理什么 | 载体 | 确定性 |
|---|---|---|---|
| 模板装配 | 结构确定的页面 | 参数化模板 + 组件槽位 | 100% |
| 能力运行时解释 | 业务逻辑 pipeline | 能力运行时直接执行 | 高，可回放 |
| Code Agent | 模板覆盖不到的定制 | codegen | 概率性，兜底 |

Code Agent 从"唯一出口"降级为三路之一，failure blast radius 收缩到定制部分。

### 3.3 先裸服务，后 Skill 封装

#### service / skill / runtime 是三层，不是三个阶段

先厘清一个易混点：`运行时`、`服务`、`skill` 不在同一根轴上，「第一阶段服务、第二阶段 skill」**不是同一对象换形态**。

- **能力运行时（capability-runtime，B 层）**：承载环境本身——保证可回放、管状态、管证据的那个壳子。
- **服务端点（service）**：能力对外的**调用面**。有输入 schema、输出 schema、挂证据的可调用单元。**两个阶段都在**。
- **skill**：能力对内的**封装面**。把执行计划 + 知识 + 工具绑定 + 评估 + 证据契约打成可注册、可路由、可版本化的包。**第二阶段才补上**。

service 在外、skill 在内、能力运行时是装它俩的容器，三者并存、不互相取代。

#### 两阶段演进的是封装程度，不是调用形态

不要求一开始就抽象成 skill 包。能力运行时分两阶段演进：

- **阶段 0（裸服务）**：暴露两类裸服务端点，封装面是直写函数，没有 skill 元数据。
  - **内核横切端点**（与场景无关）：数据服务、知识服务、工具服务、证据服务、评估服务。
  - **场景业务端点**（场景专属）：如商机洞察的 `validate / score / build-evidence / generate-cards`，编排时调用内核横切端点取数取证。
- **阶段 1（skill 封装）**：把场景端点背后的契约 + 知识 + 评估 + 执行顺序收敛封装成 skill 包；内核端点保持共享，**服务端点签名不变、App 零改动**——因为变的是封装面（裸函数→skill 包），不是调用面（service）。

会让人误以为「服务变成了 skill」，是因为阶段 0 能力**只**以裸服务存在（没有独立封装），「能力」≈「服务」；阶段 1 能力被正式封装，「能力」的指代才漂到 skill 上。这是叙述视角漂移，不是对象替换。

让这次升级无痛的**唯一不变量**：

> 每个服务端点从第一天起就强制：输入有 schema、输出有 schema、结论挂 evidence。

守住它，阶段 1 只是内部封装升级；破了它（自由文本、无证据），skill 化时整条调用链返工。

### 3.4 能力运行时 = 内核 + 多场景 Skill，不是通用引擎

能力运行时（B）不是一个笼统的「通用执行引擎」，而是「一个共享运行内核 + 一组可插拔的电商场景 Skill」：

- **运行内核**：一组横切服务端点——数据服务、知识服务、工具服务、证据服务、评估服务，加执行 / 回放。跨场景复用，与具体业务无关。
- **场景 Skill**：每个电商经营场景是一个独立 Skill 包，挂在内核上。当前已实现「商机洞察」，规划中有「主图策划 / 商品运营 / 新品洞察」等。

含义：扩展一个新场景 = 往内核挂一个新 Skill 包，而不是改内核或重写一套能力运行时。所谓「全场景通用」指的是**内核通用**，不是**逻辑通用**——业务逻辑永远是场景专属的、带契约和证据的 Skill。

## 4. 架构演变过程

```text
阶段A  PRD(静态) -> task-breakdown -> 模板+技术栈 -> Code Agent 一把梭 -> App   （单一出口）
   │  发现：不确定性集中、能力不可复用、沉淀方向错
阶段B  问题确诊
   │
阶段C  A/B 切分：App = 壳(A) + 能力运行时(B)            （能力成为单一真相）
   │
阶段D  多载体装配：Code Agent 从唯一出口降级为三路之一    （收缩失败半径）
   │
阶段E  先裸服务后 Skill：能力运行时 阶段0 裸服务端点 -> 阶段1 skill 封装 （演进而非抽象）
   │
阶段F  Code Agent 职责带 + 运行期禁区 + 外溢门禁          （能力可生长，边界明确）
```

每一步都在做同一件事：**把不确定性从运行期挤出去，沉淀成可验证、可复用、可回放的确定性资产。**

---

## 5. Code Agent 能力边界模型

### 5.1 边界由什么决定

常见误解：Code Agent 越强，就能做越多、最终取代模板和能力运行时。

真实判断：限制 Code Agent 的不是它的**能力强弱**，而是一个跟能力**正交**的约束——**确定性**。无论多强，它的输出本质仍是"每次可能不同"的概率产物；而模板装配和能力运行时解释要的是"每次一定相同 + 可回放 + 可审计"。能力提升解决不了确定性问题。

因此真正的分界线是产物性质，不是模型水平：

```text
                    会被固化/测试/复用的产物        被当作事实的一次性结论
                    （生成期 / 编译期 / 沉淀期）     （运行期实时）
  Code Agent 强  →   职责越大越好                 →   再强也不能交给它
```

### 5.2 边界公式

> **Code Agent 能外溢多远 = 其产物能被自动验证 / 回滚 / 审计到多远。**

边界由产物的**可验证性**决定，不由模型能力决定。这是整篇白皮书最关键的论断。

### 5.3 职责可生长的方向

Code Agent 变强后，正确的升级是往链路**上游和元层**走，而不是侵入运行期：

- **① 往左（编译）**：从写定制代码升级到承担 TDS Builder，把 PRD / 能力编译成结构化 spec。
- **② 往上（造件）**：从填兜底分支升级到**生产装配器要用的素材**——造模板、造服务端点。把今天的定制，沉淀成明天的模板。
- **③ 往后（蒸馏）**：从一次性生成升级到读运行期 evidence/trace，把散装服务端点蒸馏成 Skill bundle。这是"沉淀"里最难的智力活，最该交给强 Code Agent。

## 6. 能力外溢的前置条件（重点）

职责能外溢，但每一段外溢都有**门禁**。满足门禁才允许 Code Agent 接管，不满足则回退到默认态。这是"能力外溢"能安全发生的全部前提。

### 6.1 分段前置条件

| 职责段 | 默认态 | 外溢前置条件（满足才允许接管） | 不满足时回退 |
|---|---|---|---|
| ① 编译 spec | 人工 / 模板编译 | 输入有结构化 schema；编译产物有输出 schema 与门禁；产物作为"提议"经 review 才采纳 | 退回人工 / 模板编译 |
| ② 造模板·造端点 | 预制组件库 / 兜底定制 | 产物有契约测试；进入版本化资产库；端点强制 input/output schema + evidence；验证通过才晋升确定资产 | 退回一次性兜底定制 |
| ③ 蒸馏 Skill | 人工沉淀 | 运行期 trace/evidence/run record 完整可回放；有 eval/regression 防退化；离线产出经回归测试才上线 | 保持散装服务端点，不升级 |
| 运行期执行 | 能力运行时确定性执行 | **禁区，不可外溢** | 永远落在能力运行时 |

### 6.2 通用总闸

四条**同时成立**，才允许任何一段外溢：

1. **可验证**：产物有 schema + gate + test，能被自动判对错。
2. **可回滚**：版本化、隔离 worktree，错了能干净退回。
3. **可审计**：保留 diff / review / evidence，决策可追溯。
4. **默认提议**：Code Agent 产物默认是"提议"，经门禁与 human-confirmed apply 才采纳，绝不自动生效。

总闸的含义：**外溢的速度不该追赶模型能力的速度，而该追赶验证能力的速度。** 验证、回滚、审计建到哪里，Code Agent 的职责才允许长到哪里。

---

## 7. 运行期禁区：不可逾越的边界

无论 Code Agent 多强，以下永远禁止：

- 运行期用 LLM **实时产出被当作事实的业务结论**。
- 运行期用 LLM **替代确定性 pipeline 执行**。
- 产出**无证据**的经营结论。

原因：这三者直接摧毁可回放、证据链和单一真相，违反"LLM 不负责事实真实性""每个结论必须可追溯""Runtime 必须可回放"。

关键澄清——**蒸馏不是运行期执行**。读运行证据、离线编译出 Skill bundle，是生成期/编译期的动作，产物还要经回归测试才上线，因此它不触碰禁区。禁的是"运行时实时算事实"，不是"离线把证据沉淀成能力"。

---

## 8. MVP 1.0 形态：A+B 物理合体、逻辑分离

A/B 是逻辑边界，不强制是物理部署边界。MVP 1.0 可以让 App 壳(A) 与能力运行时(B) 打进同一个进程/仓库部署：

- **物理合体可以**：省掉跨服务网络与运维，1.0 一个部署单元即可。
- **逻辑必须分离**：业务能力从第一天起以「带 schema、挂 evidence、留 trace」的端点形态存在，UI 只调用、不内联业务逻辑。
- **结果**：1.0 是「单进程 + 模块边界清晰 + 契约证据齐全」，将来拆独立服务是搬运而非重写。

可以阶段性弱化的：闭环回写（rsi_feedback）1.0 可暂不做。
不能省的：trace / evidence——省了它，后续 distill Skill 就失去依据。

口径：**物理可以合，逻辑不能混。** 合体合的是部署形态，不是契约边界。

## 9. 生态飞轮：四角色 × 四面 Skill 工件

前面把架构的内部交代清楚了。这一节回答一个更上层的问题：为什么这套架构天然能长成生态，壁垒为什么会越积越厚。

### 9.1 角色与工件的天然对位

Skill 不是单一作者写出来的，它有四个面，正好对应电商生态里四类既有玩家：

| 角色 | 真正手里的资产 | 对应到 Skill 的哪一面 |
|---|---|---|
| 培训老师 | 系统化方法论骨架（标准化、可教学） | **骨架方法论**（Strategy IR / Workflow DAG） |
| TP 代运营商 | 跨店实操 SOP、踩坑经验、阈值经验值 | **实操层 + 圈层阈值**（Workflow 补丁 / Eval 校准） |
| 数据服务商 + 平台 | 行业大盘、竞品、流量、交易数据 | **数据供给**（Data Requirement / Tool Contract） |
| 商家 | 真实店铺经营反馈与决策 trace | **运行期 evidence**（Evidence Pack / Eval 回写） |

四角色不是被动的"合作伙伴"，而是对应了 Skill 工件链上四个本就需要作者的位置。

### 9.2 圈层分化要求 Skill 是参数化的

一个商机洞察 Skill 给雪花膏小店和高客单美妆品牌用，判定阈值和数据口径不可能一样。所以 Skill 必须是「**骨架方法论 + 圈层参数**」的两层结构，每个 Skill 在不同圈层下展开成不同变体。这反过来给四角色各自一个明确贡献位置：老师贡献骨架，TP 贡献阈值，数据服务商提供切片维度字段，商家用 evidence 校准变体。

### 9.3 飞轮六圈

每一圈的转动，都让下一圈更省力：

1. **老师入驻**：把方法论编译成裸 Skill，自带学员/受众完成冷启动。
2. **商家试用**：在生成 App 里跑业务，产出运行期 evidence/trace（按 Skill 维度聚合、匿名化）。
3. **TP 校准**：TP 看到 evidence 缺口，贡献 SOP 补丁与圈层阈值，Skill 准度提升。
4. **数据服务商接入**：调用量起来后，数据服务商按 Tool Contract 把数据接进来，按调用计费。
5. **圈层切片爆发**：数据维度丰富后切片变细，一个骨架长出 N 个变体，原作者复利 N 倍。
6. **更多老师入驻**：Skill 分润透明、复利可见，吸引更多老师贡献骨架——回到第 1 圈，但起点更高。

### 9.4 三道叠加的护城河

壁垒不在功能，在网络效应：

- **Skill 密度护城河**：骨架/补丁越多，业务场景覆盖越密，竞品要追赶得补齐整张网。
- **Eval 校准护城河**（最厚）：每个 Skill 跑越多次，参数校准越准；只能靠时间积累，复制不动。
- **圈层切片护城河**：切片越细，变体越多，新入场者只能做"通用版"打不过"精细版"。

### 9.5 冷启动卡点

飞轮转起来需要三件事不能破：

1. **冷启动靠老师，不靠商家**——商家没动力试用没人背书的裸 Skill，老师自带受众能强行启动。
2. **运行期 evidence 必须按 Skill 维度聚合 + 匿名化**——否则反馈回不到上游作者，第 3 圈转不动。
3. **trace 必须资产标签可归属**——每次调用要能算清楚分润给老师/TP/数据服务商各自多少，否则没人愿意贡献。

### 9.6 架构对位

这套生态不需要重写架构，现有工件链 + 能力运行时**就是开放面**。需要做的只是把内部门变成外部门：

- document-to-skill 编译器 → 外部 SDK / 老师上传入口
- Tool Contract / Data Requirement → 数据服务商注册中心
- Skill Spec 加圈层切片元数据 → TP 补丁通道
- 运行期 trace / evidence → 按 Skill 聚合的反向回灌通道

也就是说：第 0 天每个工件就被设计成 schema-first、契约清晰、可回放，**就已经把生态接口预留好了**——开放面已经在那儿，只需要打开。

---

## 10. 落地不变量清单

无论实现怎么演进，这几条不动：

1. App 壳与能力运行时在职责上分离；壳可漂移，能力单一真相。
2. 每个服务端点：输入有 schema、输出有 schema、结论挂 evidence。
3. 能力运行时先以裸服务上线，端点签名稳定后再内部封装为 Skill。
4. Code Agent 产物默认是提议，经"可验证 / 可回滚 / 可审计 / human-confirmed apply"四闸才采纳。
5. 运行期的事实计算与执行永远落在能力运行时，不交概率组件。
6. 沉淀走运行期 evidence/反馈回写能力运行时，不从 app 代码逆向 distill。
