业务需求到应用生成全链路

业务需求先整理成产品需求，再补充业务能力、应用形态和验收标准，形成 TDS Bundle（应用生成输入包）；应用装配器用模板、Runtime 绑定和 Code Agent 定制生成三种方式产出 App 壳，运行反馈再写回共享 Runtime。

BRD PRD Capability Context AppSpec EvalSpec TDS Bundle（应用生成输入包） 应用装配器 App 壳 A Runtime B

极简架构图

主线：BRD -> PRD；PRD + Capability Context + AppSpec + EvalSpec -> TDS Builder -> TDS Bundle（应用生成输入包）-> 应用装配器（三路分流）-> App 壳 (A)；运行期 App 壳调用共享 Runtime (B)，证据反馈必须写回 Runtime。

模块说明

BRD / PRD

BRD 定义业务目标和成功指标；PRD 把业务需求产品化为流程、功能范围和验收标准。

Capability Context

业务能力上下文。补充数据、知识、工具、样例、反例和风险，是能力投影的来源。

AppSpec

应用规范。定义模板、UI、技术栈、组件槽位、运行边界和功能设计。

EvalSpec

评估规范。定义质量门禁、评分维度、验证命令、安全规则和风险检查。

TDS Builder

编译器。把 PRD、Capability Context、AppSpec、EvalSpec 合成 TDS Bundle（应用生成输入包）。

TDS Bundle（应用生成输入包）

单一输入包。把产品目标、页面范围、业务能力、数据字段、验收标准和验证方式整理成可执行说明。

应用装配器

应用生成入口。按确定性分流：模板装配做结构，Runtime 绑定接能力，Code Agent 做定制。

App 壳 (A)

用户可见应用。负责页面、交互、状态和预览，可以重生成，不承载长期业务事实。

共享 Runtime (B)

业务能力运行实体。负责输入校验、规则计算、流程执行、工具调用、证据追踪和结果回放。

证据反馈

运行结果、评估记录和用户修正必须写回 Runtime，让能力持续校准并改进后续生成。

Skill Pack

稳定能力的可选沉淀形态。一阶段可以先不抽象成 Skill Pack，等 Runtime 跑稳后再沉淀契约、知识、工具和 eval。

Code Agent

定制生成分支。只处理模板和 Runtime 绑定覆盖不到的特殊交互、特殊集成或代码改造。

商机洞察例子

需求到产品

BRD 是“销售团队要从客户数据里发现高潜商机”；PRD 落成上传数据、查看机会看板、打开机会详情三类用户流程。

TDS Bundle（应用生成输入包）

写清楚要用哪些字段、生成哪些页面、调用哪些能力、如何验收：样例数据能跑通，每条推荐都有评分理由和证据。

应用装配器

上传页和看板走模板装配；评分、分层、洞察卡片是 Runtime 通用能力在商机场景里的投影；特殊交互再交给 Code Agent。

App 壳 (A) + Runtime (B)

App 壳是商机洞察工作台；Runtime 的输入校验、规则计算、结果生成、证据回放，在这里落成数据清洗、机会评分和洞察理由。

换句话说，商机洞察 App 可以重生成；可复用的是 Runtime 里的通用能力结构，以及 Skill Pack 里的场景规则、证据口径和评估用例。

设计原则

需求先产品化

BRD 先变成 PRD，业务语言先转成产品语言。

输入先编译

PRD、能力、应用和评估先进入 TDS Builder 统一对齐。

生成三路分流

确定结构走模板，业务逻辑走 Runtime，特殊定制才走 Code Agent。

壳与能力分离

App 壳可替换，业务能力在共享 Runtime 中保持一致。

证据回写沉淀

反馈必须先写回 Runtime；是否沉淀为 Skill Pack 可以后置。

一句话：TDS Bundle（应用生成输入包）统一生成上下文，应用装配器生成 App 壳，Runtime 承载通用能力并接收运行反馈。
