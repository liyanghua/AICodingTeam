# 右侧 Agent 数据需求映射产品规范

## 状态

本文档定义 `report_generator` 生成应用中右侧 Agent 的数据需求映射交互规范。当前目标是先把“业务文档描述如何映射到后台数据能力”固定为可 review 的产品契约，后续再按该契约升级应用代码。

本文档不改变节点执行语义，不启用默认 live 取数，不修改 `tasks/current/*`。右侧 Agent 仍是协作层；中间节点区、节点产物和 run artifacts 才是事实源。

## 背景

市场洞察类业务文档通常以自由文本描述数据需求，例如“获取类目排行和商品排行，生成行业前 300 商品分析表”。这类描述无法直接等同于一个后台 API 调用，因为中间至少需要三次转译：

1. 理解用户业务输入和当前节点上下文。
2. 将业务数据需求映射到可用数据能力或 API。
3. 将业务字段、口径和产物 schema 映射到 API 的请求参数和响应字段。

如果右侧 Agent 只提供“查接口”或“拉样例数据”按钮，用户会遇到 `api_id_required`、字段缺失、参数不清楚等不可行动状态。正确的产品形态应是人和 Agent 协同完成一份可确认的数据映射合同，再决定是否尝试样例取数。

## 产品目标

右侧 Agent 的目标不是替代用户判断，也不是自动完成数仓查询，而是帮助用户把自由业务描述沉淀为结构化、可追溯、可确认的数据需求映射。

成功状态：

- 用户能看到当前节点的数据需求来自哪段业务文档、哪个节点视图和哪些上游产物。
- 用户能确认 Agent 理解出的类目、周期、产品线、目标、数据口径是否正确。
- 用户能看到候选 API、工具链、参数缺口、字段匹配和风险。
- 用户能人工确认 API、字段、缺口和替代口径，形成 `data_mapping_contract`。
- 只有合同确认后，系统才允许把结果保存为本节点输入/证据；只有显式开启 live probe 后，系统才尝试样例取数。

## 适用范围

P0 适用节点：

- `collect_top_products`
- 含 `category_top_products_300` 数据需求的节点
- 其他明确需要从业务文档描述映射后台数据能力的 data 节点

P0 不做：

- 不默认调用真实数仓。
- 不保存或展示 `.env`、token、cookie、数据库密码。
- 不让 Agent 自动得出业务结论。
- 不把未确认的候选 API 或字段匹配写成节点事实。
- 不引入完整 PI 对话 Agent；第一版仍基于现有 `db_archaeologist` bridge。

## 角色与责任

### 用户

用户负责确认业务含义、参数口径、字段映射和可接受替代方案。用户可以补充缺失信息，例如类目 ID、日期范围、价格带口径、平台或榜单类型。

### 右侧 Agent

Agent 负责读取当前节点上下文、上游产物和用户输入，提出结构化建议：

- 业务输入理解。
- 候选数据能力/API。
- 请求参数映射。
- 响应字段映射。
- 缺口、风险和下一步问题。

Agent 不直接修改节点事实，不默认 live probe，不把样例数据解释成最终业务结论。

### 后台数仓助手

数仓助手只作为工具能力来源，提供 API catalog、tool selector、asset card、field schema 和可选样例探针。其结果必须经过右侧 Agent 归一和用户确认后才进入节点证据。

## 交互流程

### 1. 理解业务输入

触发入口：右侧 Agent 面板的“理解业务输入”。

输入来源：

- 当前节点的 `business_context` 和 `node_execution_view`。
- 当前节点的 `input_model.required_data`。
- 上游中间产物，例如第一步保存的《市场洞察项目定义表》。
- 用户在右侧 Agent 输入框补充的自然语言说明。

输出展示：

- 已识别业务目标。
- 已识别参数，例如 `分析类目`、`分析周期`、`分析产品线`、`当前目标`、`目标价格带`。
- 来源引用，例如上游 artifact、业务文档流程片段、节点字段。
- 待用户确认或补充的问题。

用户动作：

- 确认理解正确。
- 修改参数。
- 补充缺失参数。
- 标记某个参数暂不可用。

### 2. 映射数据能力/API

触发入口：右侧 Agent 面板的“映射数仓 API”。

系统行为：

- 调用数仓助手的 catalog/tool selector。
- 用当前节点标题、业务片段、数据需求 ID、产物 schema 和已确认业务输入构造查询任务。
- 返回候选 API 或候选工具链。

输出展示：

- 候选 API 名称、ID、domain、capability、quality score。
- 推荐理由。
- 必填参数和缺失参数。
- 风险和不推荐原因。
- 下一步需要用户回答的问题。

用户动作：

- 选择候选 API。
- 要求换一个 domain 或能力范围。
- 补充缺失参数。
- 标记候选不可用并要求重新查找。

### 3. 匹配字段

触发入口：右侧 Agent 面板的“匹配字段”。

系统行为：

- 读取选中 API 的 asset card。
- 将节点所需业务字段与 API request/response schema 匹配。
- 标出自动匹配、低置信匹配、未匹配字段。

输出展示：

- 请求参数映射表：业务参数、API 参数、来源、状态。
- 响应字段映射表：业务字段、API 字段路径、类型、描述、置信度、匹配依据。
- 未匹配字段和建议替代字段。
- 需要用户确认的口径差异。

用户动作：

- 确认某个字段匹配。
- 手动选择 API 字段。
- 接受替代字段。
- 标记字段暂缺，后续人工补充。

### 4. 人工确认并形成合同

触发入口：右侧 Agent 面板的“保存为本节点输入/证据”或后续“确认映射合同”按钮。

确认内容：

- 业务需求是否正确。
- API 是否选定。
- 请求参数是否足够。
- 响应字段是否能覆盖节点产物 schema。
- 未匹配字段是否有处理方案。
- 是否允许后续样例取数。

保存结果：

- 形成 `data_mapping_contract`。
- 写入本节点 evidence/artifacts。
- 节点状态可以变为“待取数”或“数据映射已确认”，但不得伪装成真实数据已完成。

### 5. 可选样例取数

触发入口：右侧 Agent 面板的“拉取样例数据”。

前置条件：

- `data_mapping_contract.status` 为 `confirmed`。
- 已选定 `selected_api.api_id`。
- 必填请求参数已填写或明确标记为可默认。
- 环境变量 `DBA_LIVE_PROBE=1`。

输出展示：

- 样例探针状态。
- 返回行数和字段。
- 请求参数摘要。
- 证据引用。
- 不足以作为业务结论的提示。

如果 live probe 未开启，必须显示明确阻断原因和下一步操作，而不是显示 raw error。

## 数据映射合同

`data_mapping_contract` 是右侧 Agent 数据协作的核心产物。它记录“业务需求如何映射到数据能力”的可确认结果，而不是一次工具查询日志。

最小字段：

```json
{
  "node_id": "collect_top_products",
  "business_requirement": {},
  "source_context_refs": [],
  "known_params": {},
  "candidate_apis": [],
  "selected_api": {},
  "request_param_mapping": [],
  "response_field_mapping": [],
  "unmatched_fields": [],
  "human_decisions": [],
  "evidence_refs": [],
  "status": "draft"
}
```

状态枚举：

- `draft`：已开始理解，但还没有可确认建议。
- `suggested`：Agent 已给出候选 API 或字段匹配。
- `needs_input`：缺少用户参数、API ID、字段口径或权限配置。
- `confirmed`：用户确认映射合同，可作为本节点输入/证据。
- `rejected`：用户拒绝当前建议，需重新映射。
- `sample_ready`：已在确认合同基础上生成样例取数证据。
- `blocked`：工具、权限或必要配置阻断。
- `degraded`：外部 spec-pack 未配置或不可用，但基础应用不受影响。

## UI 约束

- 右侧 Agent 面板必须独立于中栏节点详情，不得与中栏内容混排或互相挤压。
- 当前节点业务片段、上游产物、候选 API、字段映射和人工确认区必须分区展示。
- 长 API ID、字段路径、错误信息和 JSON 片段必须可换行或内部滚动。
- 缺参数、缺字段或外部工具未就绪时，优先展示“下一步要补什么”，不得只显示 raw error。
- “拉取样例数据”必须在合同未确认或 live probe 未开启时给出清楚的不可用状态。
- 保存按钮文案必须区分“保存映射合同/证据”和“保存真实数据产物”。

## 安全边界

- 默认 `DBA_LIVE_PROBE` 关闭。
- 未开启 live probe 时不得调用真实取数工具。
- 不读取、不保存、不展示 `.env` secrets、API key、cookie、数据库连接串或完整进程环境。
- `DB_ARCHAEOLOGIST_SPEC_PACK` 只作为本地外部能力引用，不复制 spec-pack 内容进生成应用。
- 样例数据只能作为证据或草稿产物，不自动生成业务结论。
- 未确认的 Agent 结果不得进入节点事实源。

## 验收标准

P0 人工验收：

- 第二步节点能从第一步《市场洞察项目定义表》提取类目、周期、产品线。
- 点击“理解业务输入”后，右侧 Agent 展示可确认业务输入和来源。
- 点击“映射数仓 API”后，右侧 Agent 展示候选 API、参数缺口、风险和下一步问题。
- 点击“匹配字段”后，右侧 Agent 展示业务字段到 API 字段的匹配表。
- 未确认字段前，系统不自动拉样例数据。
- 未开启 `DBA_LIVE_PROBE=1` 时，样例取数显示明确阻断原因。
- 保存为本节点输入/证据时，保存的是 `data_mapping_contract` 摘要和 evidence ref，而不是不可解释的原始工具 JSON。

回归验收：

- 右侧 Agent 未配置外部 spec-pack 时应用仍可使用。
- `appcheck config` 和 `runtime_smoke.js` 不依赖真实数仓凭据。
- 生成应用的右侧 Agent 与中栏节点详情保持布局独立。
