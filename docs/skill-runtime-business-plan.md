# 业务需求到应用生成：最小闭环与生态飞轮商业策划书

> 面向公司内部所有角色的共识版本。目标不是先做一个复杂生态，而是先跑通一个可验证、可复用、可商业化的最小闭环，再逐步引入外部角色放大飞轮。

---

## 1. 一句话

我们要做的不是一次性生成 App 的工具，而是一个能把业务需求变成应用、把应用运行反馈沉淀回业务能力的系统。

第一阶段先跑通最小闭环：

```text
业务需求
-> 产品需求
-> TDS Bundle（应用生成输入包）
-> 应用装配
-> App 壳
-> Runtime 执行业务能力
-> evidence / feedback 回写 Runtime
-> 下一次生成与运行更准
```

未来再把稳定能力封装成 Skill Pack，引入老师、TP、数据服务商、商家等生态角色，让能力越跑越准、越用越多、越沉淀越厚。

---

## 2. 为什么先做最小闭环，而不是直接做生态

白皮书里的生态飞轮方向是对的，但它属于平台阶段，不属于冷启动阶段。

如果一开始就做老师入驻、TP 补丁、数据服务商注册、Skill 分润、Marketplace、圈层切片，会过早进入平台复杂度。真正的风险不是想象力不够，而是底层闭环还没有证明，生态接口就先铺开了。

现在最重要的不是“有多少角色”，而是证明一件事：

> 一个业务场景能否被生成成 App，并且 App 运行后产生的证据和反馈，能否持续改进 Runtime 的业务能力。

只有这个闭环成立，生态角色才有真实贡献位置。否则生态角色只是概念分工，没有商业抓手。

---

## 3. 我们真正解决的问题

当前很多 AI 应用生成系统只解决“生成一个能看的 App”，但没有解决三个更难的问题：

1. **业务能力无法复用**
   生成出来的代码和页面绑定在一起，下次换一个应用形态就要重来。

2. **运行结果无法回放**
   App 运行时用了什么数据、什么知识、什么策略、为什么得出这个结论，缺少证据链。

3. **反馈无法沉淀**
   用户说“这个判断不对”，系统不知道应该改 PRD、改数据、改策略，还是改代码。

我们的切入点是把 App 拆成两层：

- **App 壳**：页面、交互、状态、预览，可以重生成。
- **Runtime**：数据、知识、策略、工具、证据、反馈，是业务能力的真相源。

商业价值来自 Runtime，而不是某一次生成出来的 App 外壳。

---

## 4. 最小闭环产品形态

第一阶段不做完整生态，只做一个能跑通的业务闭环。

### 4.1 输入

用户提供业务需求，经过产品化后形成 PRD。系统再补充三类上下文：

- **Capability Context**：业务能力、数据字段、知识来源、工具能力。
- **AppSpec**：页面结构、模板、交互、技术栈、功能范围。
- **EvalSpec**：验收标准、验证命令、质量门禁、风险边界。

这些内容进入 TDS Builder，形成一份 TDS Bundle（应用生成输入包）。

### 4.2 生成

应用装配器不把所有事情都交给 Code Agent，而是三路分流：

| 路线 | 处理内容 | 商业意义 |
|---|---|---|
| 模板装配 | 上传页、看板、详情页、表单、列表等确定结构 | 稳定、便宜、可规模化 |
| Runtime 绑定 | 数据校验、规则计算、结果生成、证据回放 | 能力可复用、可回写 |
| Code Agent 定制生成 | 模板覆盖不到的特殊交互、特殊集成 | 保留灵活性 |

### 4.3 运行

App 壳不直接承载业务判断。它调用 Runtime。

Runtime 负责：

- 调用数据接口。
- 调用 LLM-wiki / RAG / 知识库。
- 调用策略规则。
- 生成结果。
- 记录 evidence 和 trace。
- 接收用户反馈和评估结果。

### 4.4 回写

运行期的 evidence / feedback 必须先写回 Runtime。

第一阶段可以不抽象成 Skill Pack，但不能没有 evidence 和 trace。因为没有证据，就没有后续改进；没有 trace，就没有能力沉淀。

## 5. MVP 应该从哪里开始

建议从一个业务场景开始，例如“商机洞察”。

### 5.1 为什么选单场景

单场景更容易验证完整闭环：

- 输入是否清楚。
- 生成 App 是否可用。
- Runtime 是否能产出业务结果。
- 结果是否有证据。
- 用户反馈是否能改进下一次运行。

不要一开始追求多场景覆盖。多场景是 Runtime 跑通后的扩展结果，不是起点。

### 5.2 商机洞察 MVP

最小版本可以包含三个页面：

- 数据上传页。
- 机会看板页。
- 机会详情页。

Runtime 提供六个基础端点：

| 端点 | 作用 |
|---|---|
| `validate_input` | 检查输入数据是否可用 |
| `assemble_context` | 汇总数据、知识、策略 |
| `execute_strategy` | 执行规则、评分、排序 |
| `generate_result` | 生成可展示结果 |
| `record_evidence` | 记录证据、来源、trace |
| `collect_feedback` | 接收用户修正和评估反馈 |

这六个端点先裸服务化，不急着封装成 Skill Pack。

## 6. 数据、知识、策略如何进入 Runtime

Runtime 不应该把所有内容写死，而应该有三类 Provider。

```text
Runtime
  ├─ Data Provider
  ├─ Knowledge Provider
  ├─ Strategy Provider
  ├─ Tool Provider
  └─ Evidence Ledger
```

### 6.1 Data Provider

负责事实输入。

例如客户数据、交易数据、行业数据、竞品数据、平台数据。要求有字段契约、权限边界、版本记录和运行快照。

### 6.2 Knowledge Provider

负责知识输入。

例如 LLM-wiki、RAG、行业知识库、方法论文档、历史案例。它不直接给“最终结论”，而是给带来源的知识片段。

### 6.3 Strategy Provider

负责判断口径。

例如评分规则、优先级规则、风险判断、推荐策略。策略必须版本化，否则同一个输入下次可能算出不同结果，无法回放。

### 6.4 Evidence Ledger

负责证据账本。

每次运行都要记录：

- 用了哪些数据。
- 检索了哪些知识。
- 执行了哪个策略版本。
- 产生了什么结果。
- 用户如何反馈。
- 哪些反馈进入了改进队列。

这就是未来商业飞轮的燃料。

## 7. 商业机制设计

商业机制要分阶段设计，不能一开始就上 Marketplace。

### 7.1 阶段一：自营闭环

目标：证明客户愿意为“业务结果 + 可解释应用”付费。

收入模式：

- 项目制 PoC。
- SaaS 订阅。
- 按生成应用数量收费。
- 按 Runtime 调用量收费。

这个阶段不分润，因为生态角色还没有正式进场。

核心指标：

- 每个场景能否在 1-3 天内生成可预览应用。
- Runtime 结果是否有证据链。
- 用户反馈是否能进入改进队列。
- 第二次生成是否比第一次更准。

### 7.2 阶段二：内部能力资产化

目标：把反复出现的业务能力沉淀成内部资产。

收入模式：

- 基础平台费。
- 高级 Runtime 能力包收费。
- 行业模板收费。
- 数据接口成本转售或加价。

商业动作：

- 把高频场景沉淀成能力包。
- 把高频页面沉淀成模板。
- 把高频验证项沉淀成 EvalSpec。
- 把高频反馈沉淀成策略版本。

这个阶段可以开始出现“内部 Skill Pack”，但仍然不急着开放生态。

### 7.3 阶段三：受控生态

目标：让外部角色贡献能力，但必须经过契约、评估和证据门禁。

可引入角色：

| 角色 | 贡献什么 | 收益方式 |
|---|---|---|
| 方法论作者 / 老师 | 场景方法论、流程骨架、判断框架 | 能力包订阅分成 |
| TP / 代运营 | 实操 SOP、行业阈值、圈层经验 | 策略补丁分成、效果服务费 |
| 数据服务商 | 行业数据、竞品数据、平台数据 | 数据调用费、数据包订阅 |
| 商家 / 客户 | 真实运行反馈、修正样例、业务证据 | 更准的应用、更低的服务成本 |

平台负责：

- 审核能力资产。
- 记录调用归因。
- 计算分润。
- 做质量排序。
- 下架低质量资产。

### 7.4 阶段四：开放 Marketplace

目标：让能力资产形成网络效应。

这时才适合做：

- Skill marketplace。
- 数据服务商注册中心。
- TP 补丁市场。
- 方法论作者后台。
- 分润账单。
- 资产排行榜。
- 行业圈层切片。

## 8. 生态角色如何逐步引入

生态角色不是一起上，而是按瓶颈引入。

### 8.1 什么时候引入方法论作者

当多个客户反复需要同一套业务判断框架时，引入方法论作者。

判断信号：

- 客户问的是“怎么判断”，不是“怎么生成页面”。
- 同一类方法论被多次复用。
- 内部团队维护规则成本开始变高。

### 8.2 什么时候引入 TP / 代运营

当通用方法论不够准，需要圈层经验和阈值校准时，引入 TP。

判断信号：

- 不同行业、不同规模客户反馈差异很大。
- 评分规则需要细分。
- 客户说“这个逻辑大体对，但不适合我这个圈层”。

### 8.3 什么时候引入数据服务商

当结果质量的瓶颈变成数据覆盖，而不是算法或页面时，引入数据服务商。

判断信号：

- Runtime 经常因为缺数据无法判断。
- 客户愿意为更高质量数据付费。
- 数据调用可以被证据链清楚归因。

### 8.4 什么时候引入商家反馈机制

从第一天就要引入，但一开始只做内部回写，不做公开生态贡献。

判断信号：

- 用户反馈能稳定映射到策略调整、样例补充、评估用例。
- 反馈能匿名化和聚合。
- 反馈能提升下一次运行质量。

---

## 9. 生态飞轮如何形成

真正的飞轮不是“角色多”，而是“每次运行都让能力资产变好”。

```text
更多客户使用 App
-> Runtime 产生更多 evidence / feedback
-> 能力规则、样例、Eval 更准
-> 生成出来的 App 更好用
-> 更多客户愿意使用
-> 外部作者、TP、数据服务商有动力贡献
-> Runtime 能力更丰富
-> 平台壁垒更厚
```

飞轮有三个核心资产：

1. **运行证据**
   谁用了什么数据、什么知识、什么策略，得出了什么结果。

2. **反馈样例**
   用户为什么接受、修改或否定某个结果。

3. **评估用例**
   哪些案例可以用来判断能力是否真的变好。

没有这三个资产，生态不会形成真实飞轮，只会形成内容市场。

## 10. 公司内部各角色怎么理解

### CEO / 战略

这不是单个 App 生成器，而是从应用生成切入的业务能力平台。短期卖应用生成效率，中期卖可复用 Runtime 能力，长期卖能力生态。

### 产品

不要先做大而全平台。先把一个场景的输入、生成、运行、反馈闭环做顺。产品体验重点是让用户看懂：我输入什么、系统生成什么、结果为什么可信、反馈如何生效。

### 工程

第一天就要守住逻辑边界：App 壳可以重生成，Runtime 必须有契约、证据和 trace。MVP 可以单进程部署，但 UI 不能内联业务逻辑。

### 算法 / AI

LLM 不直接负责事实结论。LLM 可以做知识检索、解释生成、离线蒸馏、策略建议，但运行期结论必须挂数据、策略和证据。

### 数据团队

数据接口不是简单接表，而是 Runtime 的事实供给层。每次数据调用都要能被记录、回放和归因。

### 销售

不要卖“AI 生成 App”这个功能点，要卖“从业务需求到可运行应用，再到业务能力持续变准”的结果。

### 客户成功

客户反馈不是售后问题，而是 Runtime 的训练燃料。要把反馈收集成结构化样例，而不是散落在聊天记录里。

### BD / 生态

前期不要急着拉生态伙伴。先观察闭环里真正的瓶颈在哪里：缺方法论、缺实操阈值，还是缺数据。瓶颈出现后再引入对应角色。

### 财务

短期收入看项目和订阅，中期看 Runtime 调用和能力包，长期看 marketplace 抽成和数据调用分成。

### 法务 / 合规

重点关注数据权限、反馈匿名化、证据归属、第三方能力责任边界和分润归因。

## 11. 关键指标

### 11.1 最小闭环指标

| 指标 | 说明 |
|---|---|
| 生成成功率 | PRD 到可预览 App 的成功比例 |
| 首次可用时间 | 从需求输入到可预览 App 的时间 |
| Runtime 调用成功率 | 业务能力端点是否稳定 |
| Evidence 完整率 | 结果是否都有证据链 |
| 反馈回写率 | 用户反馈是否进入 Runtime |
| 二次改进率 | 反馈后下一次结果是否改善 |

### 11.2 商业指标

| 指标 | 说明 |
|---|---|
| 单客户场景数 | 一个客户是否从一个场景扩到多个场景 |
| Runtime 调用量 | 能力是否被持续使用 |
| 能力复用率 | 同一能力是否被多个 App 复用 |
| 人工替代时长 | 是否减少产品、运营、开发时间 |
| 客户续费意愿 | 是否愿意为持续改进付费 |

### 11.3 生态指标

| 指标 | 说明 |
|---|---|
| 外部能力通过率 | 外部角色贡献资产是否过质量门禁 |
| 能力调用归因率 | 是否能算清谁贡献了价值 |
| 分润准确率 | 是否能透明结算 |
| 圈层变体数量 | 一个方法论能否扩展到多个细分场景 |

## 12. 90 天落地计划

### 第 1-30 天：跑通单场景闭环

- 选择商机洞察作为样板场景。
- 固定 3 个页面模板。
- 建立 6 个 Runtime 裸服务端点。
- TDS Bundle 能描述页面、能力、数据、验收。
- 每次运行生成 evidence 和 trace。

交付物：

- 一个可演示 App。
- 一个 Runtime run record。
- 一个 feedback 回写队列。

### 第 31-60 天：证明反馈能改进结果

- 收集真实或半真实用户反馈。
- 把反馈转成策略调整、样例补充、Eval 用例。
- 对比第一次和第二次运行结果。
- 建立最小评估看板。

交付物：

- 反馈样例库。
- EvalSpec 最小集。
- Runtime 能力版本记录。

### 第 61-90 天：复制到第二个场景

- 选择相邻场景，例如新品洞察或商品运营。
- 复用同一套 Runtime 内核。
- 新增少量场景端点。
- 验证是否能复用数据、知识、策略、模板。

交付物：

- 第二个 App。
- 第一批可复用 Runtime 能力。
- 是否进入 Skill Pack 封装的判断报告。

## 13. 阶段门槛

不要靠感觉进入下一阶段，要靠门槛。

### 可以进入“内部能力资产化”的条件

- 至少一个场景跑通 20 次以上。
- 80% 以上结果有完整 evidence。
- 用户反馈能稳定映射到 Runtime 改进项。
- 同一能力被至少两个 App 或两个客户复用。

### 可以引入外部贡献者的条件

- 能力资产有 schema、版本、评估用例。
- 外部贡献可以被测试。
- 调用归因可以被记录。
- 低质量贡献可以被下架。

### 可以做 Marketplace 的条件

- 有稳定调用量。
- 有多类外部资产。
- 有分润结算能力。
- 有质量排序和治理机制。
- 有明确的客户付费意愿。

## 14. 主要风险与应对

### 风险一：过早平台化

表现：角色很多、概念很完整，但没有一个闭环真正跑通。

应对：前三个月只做单场景最小闭环，不做 marketplace。

### 风险二：App 和 Runtime 边界混乱

表现：业务逻辑又写回 App 代码里，能力无法复用。

应对：UI 只能调用 Runtime，不能内联事实判断逻辑。

### 风险三：无证据结论

表现：结果看起来很智能，但无法解释、无法回放。

应对：所有 Runtime 结果必须有 evidence 和 trace。

### 风险四：反馈无法结构化

表现：客户说了很多意见，但系统无法吸收。

应对：反馈必须归类为数据问题、知识问题、策略问题、体验问题或评估问题。

### 风险五：分润机制过早复杂

表现：还没有稳定调用量，就开始设计复杂结算。

应对：阶段一不分润，阶段二内部资产化，阶段三才做受控分润。

---

## 15. 最终商业判断

这个方向的关键不是“能不能生成 App”。生成 App 只是入口。

真正的商业价值在于：

1. 每个 App 都把一个业务场景带进 Runtime。
2. 每次 Runtime 运行都产生证据和反馈。
3. 每次反馈都让能力资产变准。
4. 能力资产越准，生成出来的 App 越可信。
5. App 越可信，客户越愿意把更多场景交给平台。
6. 场景越多，生态角色才有贡献和分润空间。

所以公司要记住一个顺序：

```text
先闭环
再资产化
再引入生态
最后做平台
```

最小闭环是商业起点，生态飞轮是放大器。没有闭环，生态是负担；有了闭环，生态才是壁垒。
