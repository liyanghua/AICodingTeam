三条链路的协作模式

三条编译链路纵向分成三个交付组，各有唯一负责人（DRI）；业务专家、产品、AI 应用开发工程师三种角色横切覆盖三条线。右侧的共享节点契约层不属于任何一组，由三位 DRI 组成的契约委员会治理——它是三条线唯一的强耦合点，也是这套分组能否成立的关键。

治理拓扑：三组如何通过契约委员会串联

单一变更入口是这张图的重点：任何组想加节点类型或改契约字段，都不能在自己代码里本地改，必须提 change request 到委员会，三方评审通过后统一发版。绕过这条路径，三条线就会各自定义一份、契约漂移、同时崩。

三种角色稀缺，不必每组硬配齐三人。每个角色在主场是 owner、在客场是 reviewer / contributor；业务专家横切保证业务真相不失真，产品横切保证 HITL 体验一致，AI 工程师横切保证契约可落地。

工作节奏：契约先行 → mock 并行 → 按契约联调

0

契约冻结

委员会先把 NodeSpec/NodeContext v0 冻结：类型清单、契约字段、HumanGate 规范。这是唯一必须最先对齐的强耦合点。

1

mock 并行

三组按契约各自 mock 上下游，独立开发互不阻塞：Lane1 mock 资产、Lane2 mock 输入、Lane3 mock 运行图。

2

按契约联调

只在契约边界对接，不看彼此内部实现。数据依赖是 Lane1→Lane2→Lane3，但 mock 让三组能同时开工。

3

CR 迭代

契约变更集中走 change request，委员会评审发版后三组各自跟进升级，避免散落式私改。

判别信号与反模式

契约只定义一次

新增一种节点类型，只改共享契约，Lane2 执行与 Lane3 渲染自动对齐，无需两处同步。

DRI 拍板不含糊

每条线有唯一负责人，跨线分歧升级到契约委员会，而不是无休止拉群讨论。

反模式：本地私改契约

只要有第二处节点类型定义存在，这套分组就退化成三个各自为战的系统——契约漂移、三条线同时崩。
