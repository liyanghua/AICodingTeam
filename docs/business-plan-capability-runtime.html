商业策划书 · 能力运行时（capability-runtime）

从最小闭环到生态飞轮

这是一份让公司每个角色都能看懂的策划书。它要回答三个问题： 我们到底卖什么？最小可运转的生意长什么样？未来怎么从一门生意长成一个生态。

0一句话讲清楚我们卖什么

我们把"商家做生意需要的方法论 + 数据 + 工具 + 经验"打包成一个个场景应用， 装进一个统一的能力运行时里。商家不再装一堆 SaaS，而是用一个个"能直接干活、有数据、有结论、有证据"的经营应用。

对商家

开箱即用的经营应用。不是工具，是"已经会做这件事的同事"。

对内容方 / 老师

方法论第一次能变成可调用、可计费、可复利的资产，不再只是课程和 PPT。

对公司自己

每多一个场景应用，跨商家复用；每多一个商家，反哺校准。一份资产，多次收钱。

术语对齐：对外、对客户、对销售、对财务都统一叫"场景应用"（一个应用 = 一件能干完的业务事）。 内部工程口径里它叫 Skill，是同一个东西的工程称呼。本策划书除非讨论工程资产池，否则一律用"场景应用"。

1反共识：为什么不一上来就做生态

生态图很漂亮：老师、TP、数据服务商、商家四方齐转。但四方互为前置，谁都不肯先进场。 做生态的公司，没有一家是"先画飞轮、再填角色"—— Salesforce 单边 SaaS 做了 6 年才有 AppExchange，Shopify 做了 3 年才有 App Store，HubSpot 也是先有几万付费客户才开 Academy。

陷阱 1：四向冷启动死锁

老师等用户，商家等可信策略，TP 等标准化场景，数据方等调用量。四方相互等待。

陷阱 2：放大器先于信号源

TP 和数据服务商是规模化阶段的放大器。先于"商家愿意付费"出现，等于让扩音器先响。

陷阱 3：护城河来自单一回路

真正的壁垒是"商家用 → 反馈 → 场景应用校准"这一环。只要它转起来，其他角色会自己出现。

因此本策划书的主张是：最小闭环只激活两个角色（内容方 + 商家）。生态视图保留为终局图，不作为 MVP 路线图。

2最小闭环：双边回路

MVP 只做一件事：让"一个内容方 + 一小批种子商家"围绕一个场景应用转起来。 数据自取，校准手工，TP 和数据服务商完全不上场。

① 内容方做什么

把一份业务方法论（例：电商机会洞察）编译进一个场景应用——给出步骤、阈值、所需数据、所需工具、好坏样本。

② 运行时做什么

把场景应用装配成一个真能跑的产品：拉数据、跑工作流、出结论、留证据。商家直接用，不接触底层。

③ 商家做什么

用，付费，给出"这条结论我信 / 不信、我会改成什么"的反馈。这是场景应用持续校准的唯一原料。

最小闭环的成功定义：一个场景应用，5–10 家付费商家，30 天复用率 ≥ 60%，校准能让下一个商家少改 30% 的参数。 这三件事同时达成，才有资格进入下一阶段。

3商业机制：钱怎么进、怎么分、怎么沉淀

最小闭环阶段只有两种钱，结构简单。生态阶段再分润给老师、TP、数据方。 每条规则都用一句话，让财务、销售、产品都能记住。

3.1 MVP 阶段：单向计费

收入项定义单价示意谁卖给谁
场景应用订阅费商家按月订阅一个场景应用（如机会洞察）。¥ 2k–8k / 月 / 应用公司 → 商家
上线服务费首次接入时的圈层校准 + 老板培训。一次性。¥ 5k–20k / 商家公司 → 商家

MVP 阶段内容方是自营或合作伙伴包月，不做按调用分润。理由：数据样本太小，分润算不准，会消耗信任。

3.2 生态阶段：三向分润

角色贡献物分润形式触发条件
老师 / 内容方方法论骨架 + 工作流 + 校准规则该场景应用订阅收入按比例分成（如 15–25%）该场景应用通过校准基线后
TP 代运营圈层 SOP 补丁、阈值校准、培训交付所推荐商家订阅费返点（如 20–30%，前 12 个月）商家由该 TP 上线 + 30 天留存
数据服务商数据源 + 标准接口实现调用次数计费 + 平台抽成（如 70/30）接口通过质量校验进入注册中心
商家反馈样本非现金：订阅折扣 / 优先功能 / 案例曝光反馈被采纳进场景应用校准

3.3 沉淀资产：每一笔交易都长在公司账上

场景应用模板池

方法论 + 校准参数。订阅越多越准，越准越好卖。（工程口径：Skill 资产池）

反馈证据池

每次运行留下的输入 / 中间结果 / 结论 / 反馈。是校准的唯一原料，也是壁垒。（工程口径：Evidence Pack）

数据接口标准池

统一接口。数据服务商和内部工具都按它接入，越多接入越难绕开。（工程口径：Tool Contract）

4从最小闭环到飞轮：四阶段

每个阶段都有明确的"进入条件"——上一阶段没达标就不进。这条规则保护团队不被生态愿景拖空现金。

阶段关键问题核心动作进入下一阶段的硬指标
0 商家会不会为场景应用付费？ 自营 1 个场景应用，签 5–10 家种子商家，手工校准 付费续约率 ≥ 60%；同一应用跨商家复用率 ≥ 60%
1 场景应用能不能被外部人调参？ 开放 SOP 补丁机制，签 2–3 家 TP，规模化到 30–50 家商家 TP 自交付 NPS ≥ 40；非自营圈层校准准确率 ≥ 自营 80%
2 数据接口能不能被第三方实现？ 开放接口标准注册中心，接 3–5 家数据服务商 30% 调用量来自第三方接口；接口失败率 ≤ 2%
3 第三方老师能不能进场赚钱？ 开放方法论 → 场景应用的编译入口，建立分润、准入、争议机制 第三方场景应用 ≥ 公司自营数量；老师月分润 ≥ 50k
节奏纪律：阶段没达标不进下一阶段。生态故事可以讲给资本，但内部不许跳级——跳级 = 资源被多个未验证假设同时消耗。

5飞轮演变：从单环到多边网络

第一个圈跑通后，每加入一个生态角色，"加强的是什么"不一样—— 圈 1→2 加粗原环；圈 2→3 在原环外挂一张单边数据网；圈 3→4 整体升级为双边市场。 下图把四种形态画在一起，看清楚每一步多了什么、网络性质变成了什么。

阶段 新进角色 加强的是什么 网络性质 关键参与方公式
圈 1 — 原环流程跑通 + 中心资产首次累积 还不是网络 1 内容方 × 5–10 商家
圈 2 TP 代运营 原环本身变粗：销售边 + 校准边都变粗；圈层切片让中心资产横向膨胀 规模复利
还不是网络效应		 1 内容方 × 2–3 TP × 30–50 商家
圈 3 数据服务商 在原环之外外挂一张单边数据网：接口标准成为集合点 单边网络效应
调用量 ⇄ 数据方数量		 + 3–5 数据服务商

老师 ⇄ 商家		 N 老师 × N 数据方 × N TP × M 商家

判别：你正在加强的是哪一层？

如果新角色只让"销售量 / 服务量 / 圈层覆盖"变多，那是加强原环。如果新角色多了一个，能让所有原有角色受益，那是网络效应。两者不要混。

顺序不能颠倒

原环 → 单边网 → 双边市场。颠倒了就是经典的"先建集市后开店"陷阱：双边市场必须建立在"已经有可被复制的资产 + 已经能跑通的服务流程"之上。

每一层都不消失

形态 D 不是替代形态 A，是叠加。底层的单环每天还在转——只是上面又多了两层网络在为它加速。

给所有团队成员的一句话：圈 1 验证完成 ≠ 直接进入双边市场。中间隔着"原环变粗"和"挂上单边数据网"两道台阶。每一道都要单独验证，每一道的护城河类型都不一样。

为什么"老师"才开新轴，TP 不是

多一个老师 = 平台上多一类商家能做的事。多一个 TP = 同一类事多了一个供应商。前者改变了平台的产品边界，后者只改变了交付容量。

付费社区是产能侧批发商

它本身不开新轴，但它能把"老师入驻"从一个一个谈，变成一个社区一批一批接。冷启动期建议优先合作 1–2 个头部社区作为老师 GTM 通道。

合作社区的护城河边界

反馈证据池由平台留存，不开放给社区。否则社区做大后可能反向训练自己的方案。这条要写进合作协议。

6第一性视角：AI+Agent 时代谁能建生态

AI+Agent 没有改变生态成立的底层要素，只是改变了实现成本。 用第一性原理拆三件事：生态成立靠什么、AI 改了什么没改什么、谁最有机会建起来。

6.1 生态成立的三个底层要素

可复用的资产

某种东西，能从"为 1 人服务"复制到"为 N 人服务"，边际成本递减。

可信的连接

两侧角色通过你而非彼此直接连接，且你能持续证明这个连接更划算。

可累积的反馈

每次连接都在你这里留下数据，让下一次连接的质量更高。

三者具备，生态就会自己长出来。这条规律不分时代——SaaS 时代、移动时代、AI+Agent 时代都一样。

6.2 AI+Agent 改变了什么，没改变什么

维度 传统 SaaS 时代 AI+Agent 时代
资产复制单元代码模块（要工程师写）方法论 / Workflow / Prompt（领域专家就能写）
接入新场景成本重做 UI + 后端 + 数据接入应用生成 + 适配，按天计
谁能造资产工程团队任何懂业务的人（编译器替代工程）
反馈数据壁垒用户行为日志结构化执行轨迹 + 结论 + 修正
边际服务成本人月算力 + 校准时间
内容方变现卖课、卖咨询方法论按调用分成
关键变化一句话：资产生产从"工程师瓶颈"变成"懂行人瓶颈"，懂行人池子大了 1–2 个数量级。
没变的：信任建立、付费决策、案例传播、行业 know-how 稀缺性。电商老板该慢还是慢。

6.4 七类玩家的概率排序与出发点

第一性视角：谁手里同时握住"资产 + 连接 + 反馈"三者的种子，谁概率最高。注意"握得多"不等于"握得全"——单维度极强但其他维度为零的玩家，概率反而低。

6.5 对我们的关键启示

对概率 2（社区）

最适合做 GTM 合作伙伴。有人没工程，让他们成为产能侧批发商。

对概率 3（SaaS）

最适合做退出路径或集成合作方。圈 1+2 跑通后，他们是最可能的收购方。

对概率 4（平台官方）

最大不确定性。一旦启动会用"平台规则 + 流量绑定"挤压第三方。方法论资产要保持平台无关。

时间窗口收窄到 12–24 个月，意味着两件事： (1) 不能慢——三年慢慢做 Skill 库，到时候社区/SaaS 已经合作或并购完成生态； (2) 不能贪——资源全压在"runtime 工程化 + 场景应用编译器 + 头部社区合作"三件事上。其他都是干扰项。

7护城河层级与自有模型边界

经常被问两个问题："你们做自有模型吗？" 和 "runtime 不是别人也能做吗？" 两个问题指向同一件事——把"技术资产"误当成护城河。这一节把护城河分层排清楚，顺手回答自有模型该摆在哪里。

7.2 自有模型的真实角色：成本项，不是护城河

正确摆位

把自有模型放在成本与隐私边界，不放在差异化叙事。
触发条件：调用量大到 token 费 > 自托管成本；或商家要求数据不出私域。

什么时候做

阶段 2 之后再考虑。阶段 0/1 用 API 模型把校准样本攒起来，时机到了再做窄场景蒸馏（如归因结论生成、SOP 阈值判定）。

什么时候不做

当目的是"让 BP 看上去更硬"、"显得有壁垒"、"对标某竞品"——拒绝。自训不能产生护城河，只会把现金流和团队带宽烧在最薄那层。

7.3 runtime 工程化也是同样的位置

现在能力运行时这套工程化（编排 + Tool Registry + Trace + Eval 框架），很多团队都能在 3–6 个月内做出可用版本。 开源生态、基模厂商、SaaS 公司都在补这一层。把它当护城河叙事会被很快证伪。

runtime 工程化的真实价值：它是"中层和顶层资产能否成立"的基座。没有它，Skill 跑不起来、Eval 收不到样本、圈层切片无从拟合。 所以要做、要做好、要做早——但不要把它写进护城河叙事。

7.4 这对策略的含义

资源排序

顶层 > 中层 > 底层。
反馈链路与圈层切片的工具化优先级，要高于自训模型和 runtime 性能优化。

叙事重点

对外不讲"我们自训了模型"。
讲"我们有 N 家商家 × M 个月的采纳/修改样本，校准准确率到 X%"。这是别人 12 个月内追不上的。

合作判别

合作方愿不愿意把反馈样本沉淀在我们这里，是评估生态合作的硬指标。 样本走了，护城河就跟着走。第 5 章合作社区条款里"反馈证据池由平台留存"就是这个意思。

一句话总结：真正的护城河 = 商家反馈样本 × 圈层切片密度 × 时间。 底层工程和自有模型都是必要项，但都不是答案。把资源压在能让样本最快累积的地方，才是阶段 0–2 的正解。

7.5 还有一层：组织形状（顶顶层）

前面四层都假设"我们能持续把样本和方法论灌进来"。这件事能不能持续，取决于组织本身的形状—— 谁是重力中心、谁有 status、谁有决策权、reward 怎么发。顶层资产是结果，组织形状是原因。

8风险与边界

风险 A：场景应用不可复用

每家商家都要重做一次。意味着我们卖的不是场景应用，是定制项目。退路：把"圈层切片"作为一等公民，提前埋进方法论骨架。

风险 B：反馈不回来

商家用完不给反馈，Eval 没原料，飞轮断在第一圈。对策：反馈嵌入应用本身（结论右侧"采纳/修改"按钮 + 月度复盘约定）。

风险 C：被平台数据卡脖子

底层数据依赖电商平台 API 或抓取。对策：阶段 0 控制数据面，阶段 2 才开放给数据服务商，期间持续多源化。

风险 D：内容方反水

老师拿走场景应用模板自己卖。对策：分润够厚 + 反馈证据池留在公司 + 校准准确率为护城河（老师没有商家反馈样本就没法迭代）。

风险 E：跳级建生态

因为故事好看就提前招 TP/数据 BD。对策：每季度对照阶段硬指标 review，未达标停止生态侧招聘。

风险 F：LLM 当事实源

结论无证据 = 不可信 = 商家流失。对策：架构上把 Code Agent / LLM 限制在生成期，运行期事实必须经 Tool + Evidence。

10组织形状：顶顶层护城河

技术、模型、Skill 库都会被追平。最后没法抄的是"组织形状"——重力中心是什么、谁站在最高 status、决策权在哪、reward 怎么发。 这一节回答两件事：什么形状让前面四层资产能持续灌进来，以及对外承诺如何在组织结构里兑现。

10.1 重力中心：一句话定义

公司的重力中心 = "把方法论编译成可运行的应用，让反馈样本回到平台"。 所有岗位、产品、合作、招聘叙事都围绕这一句旋转。
不是"做 Agent 平台"，不是"做 AI 编码工具"，不是"做电商 SaaS"。

组织活动按重力分层

编译器 / 校准 / 反馈链路 = 核心；runtime 工程 / 数据接入 = 支撑；销售 / GTM = 放大器。重力反向 = 团队抄方向。

对立面要说出来

不做裸 prompt 工具、不做工程师专属 SaaS、不做通用 Agent 商店、不做"模型即产品"。挑边能让对的人立刻识别我们。

叙事高度对齐 shape

讲"懂行人第一次能把方法论变成有复利的资产"。比"做 runtime"高一档，比"重塑电商行业"低一档，刚好匹配现有团队规模。

10.2 两个新岗位：组织形状里的"主角"

这两个岗位不是 PM、不是工程师、不是讲师。它们的 status 必须是公司最高一档，否则前面四层护城河没人能持续灌进来—— 就像 Palantir 的 forward deployer：别处算低 status 的活，在我们这里是中心。

岗位 在架构中位置 日常活 为什么必须 high status

Method Engineer		 document-to-skill 编译器入口；老师内容 → Strategy IR → Skill Spec 校稿 IR · 设计参数化 · 与老师共写方法论 · 跑回归 没有这岗，方法论灌不进 Skill；中层资产停滞

检验：招聘 JD、title、薪酬带宽、晋升路径里能不能找到这两类人？ 如果他们只是"工程师"或"运营"的下挂角色，组织 shape 就还在旧形状里。

10.3 emotional ↔ structural 对账表

原则：每条对外承诺，组织 / 产品 / 合同里都必须有一项结构性兑现。口头说但结构没兑现就是假承诺，老话叫"画饼"。

我们对外讲什么（emotional） 组织 / 产品 / 合同里如何兑现（structural） 现状
反馈样本是真正的护城河 "采纳/修改"按钮是产品里 high-status 入口；反馈工程化负责人直接向 CEO 汇报 待落
圈层切片是无法复制的资产 切片治理与核心算法平级；切片版本 / 回归测试有独立 owner 与季度 OKR 待落
Skill 密度是中层资产 Skill 注册 / 版本治理 / 老师校稿台是平台一等公民产品入口，不是"内部工具" 部分落到角色映射页 §3
能力运行时是基座但不是护城河 runtime 团队 ownership 高，但叙事重力不放在他们身上；资源分配按"顶 > 中 > 底" 部分落到 §7.4
权属归平台、证据池不可外带 写进商家合同 + 合作社区分润协议；产品上证据池跨租户不可见 条款层未落
我们是"懂行人的资产生产权" 招聘 JD 出现"业务方法论工程师 / Skill 校准官"；title 直接体现新岗位 未落
合作伙伴值得长期合作 分润 / Co-publish / 共享数据池等条款写进合作协议（不靠口头） 条款层未落

10.4 时间承诺最危险

"以后就大了"

scope / authority / 经济参与全靠"以后"——时间在悄悄走。对外对内都不靠它。

"现在就给"

能写进 title / 决策权 / 经济条款 / 产品入口的，当期就写。这是"被看见"，不是"被选中"。

对合作方同义

分润比例、证据池归属、续约条款写进合同，不用"长期来看会更好"代替。

一句话总结：组织形状决定前四层护城河能不能持续累积。 重力中心 = 把方法论编译成可运行应用 + 让反馈回平台；两个新岗位（方法论工程师 / 校准官）必须是最高 status； 所有对外承诺都要有一条对应的结构兑现，否则就是假承诺。

11一页纸总结

真正的护城河（顶顶层）

组织形状：重力中心 = 把方法论编译成可运行应用 + 让反馈回平台；业务方法论工程师 / Skill 校准官是最高 status；每条对外承诺必须有结构兑现。

我们在做什么

把业务方法论 + 数据 + 工具打包成场景应用，装进能力运行时，以"会干活的应用"形式卖给商家。

现在做哪一步

阶段 0：1 个场景应用 × 5–10 家种子商家 × 双向反馈。其余角色暂停。

钱怎么进

订阅费 + 校准服务费。生态分润进入阶段 1 才打开。

护城河长在哪

Evidence 数据池 + Eval 校准准确率。商家越多，下一个商家越准。
