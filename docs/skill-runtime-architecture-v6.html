从业务资料到可试用应用：App Generation V6

V6 用业务产品语言描述完整闭环：用户给一组业务资料、企业与行业数据、私有模型能力，系统先把这些输入整理成产品需求和可执行的业务运行资产，再由应用装配工作台生成一个与具体场景和业务相关的可预览应用。应用只是用户操作入口；真正的业务判断、证据、规则和复盘沉淀在共享能力运行时。

主链路：业务资料如何变成可试用应用

这张图只讲业务产品主线：用户不是“写 prompt”，而是提交业务资料、数据和模型能力；系统不是“直接生成代码”，而是先编译业务蓝图，再装配与场景相关的应用，并把运行证据沉淀回能力运行时。

细节图：应用装配工作台如何运行

细节图把“应用装配器”展开成四件事：节点执行流负责跑流程，中间产物区负责可观测，右侧 Agent 协议负责把用户语言变成结构化动作，Code Agent 负责真正生成或修复代码。

核心组件职责

业务策略文档

说明业务目标、经营方法、流程步骤、判断标准和最终输出。它决定应用为什么存在、为谁服务、解决什么场景问题。

企业 + 行业数据

包括企业内部数据、行业数据、样例数据、口径定义和可用字段。它决定应用需要哪些数据入口、校验和证据链。

私有模型能力

包括企业私有模型、领域模型、模型调用边界和适用任务。它决定哪些节点能自动分析、总结、评分或生成建议。

业务蓝图编译器

把策略、数据和模型能力整理成产品需求、业务流程、数据规范、知识规范、模型绑定、证据标准和应用配置。

应用装配工作台

它就是应用装配器：消费业务运行资产，驱动节点执行流、代码生成、发布预览和 Agent 修复。

节点执行流

面向用户展示业务步骤；内部由图执行引擎按依赖运行，记录输入、过程、输出、风险和重跑证据。

右侧 Agent 协议

把“解释这个产物”“重跑这步”“修复预览问题”等自然语言请求转成可确认、可验证的结构化动作。

能力运行时

承载业务规则、数据访问、私有模型调用、工具调用、证据、评估、回放和可复用场景 Skill，是业务判断的事实源。

MVP 取舍：哪些先人工，哪些必须突破

可先人工：资料整理

业务策略文档、行业材料、术语表和样例数据可以先由人整理成目录与 manifest，保证输入质量，不急着做全自动采集。

可先人工：数据接入

企业数据和行业数据先支持 CSV / 文件上传、字段映射和人工确认；真实数据库、BI、ERP、API 连接可以后置。

可先人工：模型配置

私有模型先通过人工配置 endpoint、模型名、适用任务和调用边界；模型路由、自动评测和成本优化可以后置。

可先人工：业务审核

PRD++、规则阈值、证据口径和最终应用快照先由业务专家确认，避免早期系统自动放大错误结论。

可先人工：发布与反馈

发布预览、验收记录、问题标注、规则提升候选可以先走人工确认，让闭环跑通并积累高质量样本。

不能人工替代：核心壁垒

业务蓝图编译、运行资产契约、应用装配器、图执行引擎、Agent 产物协议和能力运行时必须产品化，否则无法规模化复制场景。

MVP 的原则是：低频、非结构化、需要业务判断的环节可以人工；高频、可复制、决定规模化的环节必须系统化。核心壁垒不是“把页面生成出来”，而是把业务策略、数据、模型和证据编译成可执行、可回放、可复用的场景能力。

设计原则

先理解业务，再生成应用

业务策略、数据和模型能力先编译成 PRD 与运行资产，再进入应用装配，不把原文直接丢给 Code Agent。

工作台就是装配器

工作台不是只看日志的页面，而是业务步骤、产物、预览和 Agent 协作的装配现场。

场景决定应用

生成出来的是某个业务场景的应用，不是通用页面；页面、节点、数据入口和模型能力都来自业务运行资产。

可观测优先

每一步都要看到输入、执行过程、输出、风险、证据和可操作项。

Agent 有边界

右侧 Agent 理解用户意图并生成动作；复杂代码修改委托 Code Agent，所有改动落证据。

能力可复用

一次应用生成的规则、证据和调优经验，可以沉淀为下一个业务场景的能力资产。

一句总结：V6 的核心不是“从 PRD 生成页面”，而是“从业务策略、企业/行业数据和私有模型编译出业务运行资产，再用应用装配器生成对应场景的可试用应用，并把运行证据沉淀为可复用能力”。

结尾：从共识强化到反证驱动

这组对比图作为 V6 的收束：系统不是让更多资料去证明一个已经成立的大叙事，而是把业务判断拆成可被证伪、可被实验、可被修正、可被沉淀的运行资产。应用生成工作台的价值，也在于把这种反证循环产品化。

现有模式｜容易陷入

共识强化型循环

快速理解大结构之后，信息越多，反而越容易强化已有判断。

一句话解释 这个循环会让人越来越懂行业大图，但很容易错过真正决定商业成败的细节变量。

改进模式｜刻意训练

反证驱动型循环

先承认大结构只是初步判断，再用小实验找到哪里对、哪里错。

一句话解释 这个循环不是否定抽象能力，而是把抽象后的判断变成可被证伪、可被修正、可沉淀的方法。

1. 大结构只是起点 “我理解了架构”不等于“这个判断已经成立”。
2. 每个判断都要找判别变量 继续追问：到底哪个小变量会决定成败？
3. 用最小实验反证 不要用更多资料证明自己对，要用真实结果发现哪里错。

这也是 App Generation V6 的工作方式：业务资料先形成假设，节点执行流把假设变成可观察产物，可预览应用暴露真实问题，右侧 Agent 和 Code Agent 负责把问题转成修复与证据，最终沉淀为可复用能力。
