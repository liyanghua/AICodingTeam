# 修复右侧 PI 对话"答非所问"（事件协议对齐）

## 背景（已完成部分，不重做）

T1-T20 已完成：7 份 spec、PiAgentProvider/PiRpcClient 骨架、右侧对话 SSE 后端+前端、PRD 上传+节点 SSE。`.env` 凭据注入（`load_pi_env_overrides`）、`PI_DEFAULT_MODEL`/`PI_DEFAULT_THINKING` 注入命令行均已通过。唯一未通的是：真实 pi 事件解析与路由用的是一套虚构协议，导致答案文本被全部丢弃。

## 根因（已实锤）

- 前端只认归一化事件 `message_delta/tool_call/tool_result/agent_end/upstream_error`（`dashboard/app_generation.js:818-831`）。
- 翻译唯一入口是 [`growth_dev/team/pi_rpc.py`](growth_dev/team/pi_rpc.py) 的 `default_event_parser` / `_agent_event_to_stream`，它期望 `{"type":"agent_event","event_type":"message_delta","payload":{"text":...},"id":pid}`。
- 真实 pi（参考 [`store.mjs`](/Users/yichen/Desktop/OntologyBrain/PI_AGENT/db-archaeologist-pi-spec-pack/web/public/store.mjs)）每行顶层 type 直接是 `message_update`/`tool_execution_start`/`tool_execution_end`/`agent_end`/`turn_*`；答案在 `assistantMessageEvent.text_delta.delta`；工具在 `tool_execution_*`（`toolCallId`/`toolName`/`args`/`result`/`isError`）；**流事件不带 prompt id，只有 `response` 命令回执带 id**。
- `PiRpcClient._dispatch` 按 `event["id"]` 路由 → 无 id 流事件被丢弃（第二层 bug）。
- 测试 fixture 用了同一套虚构格式 → 假绿。

## 修复方案

### 1. 重写翻译层 `pi_rpc.py`（按真实协议）

`default_event_parser(line)` 顶层 type 分支改为：
- `message_update` → 读 `assistantMessageEvent`(ame)，按 `ame.type`：
  - `text_delta` → `message_delta {text: ame.delta}`
  - `text_end` → 仅当本回合未累积过该 contentIndex 的 text 时用 `ame.content` 兜底（避免与 delta 重复）
  - `thinking_delta`/`thinking_start`/`thinking_end` → **不**映射为答案；单独发 `thinking_delta` 事件（前端默认折叠/忽略），绝不进答案气泡
  - `toolcall_*` → 忽略（真正执行看 tool_execution_*）
- `tool_execution_start` → `tool_call {tool_call_id: toolCallId, name: toolName, input: args}`
- `tool_execution_end` → `tool_result {tool_call_id: toolCallId, output: result, is_error: isError}`
- `tool_execution_update` → 忽略
- `agent_end` → 取 `willRetry` + `messages[].stopReason/errorMessage` 判 error；`usage` 从顶层或 messages；stop_reason 同源
- `turn_end` → 终止兜底（无 agent_end 时）
- `auto_retry_start` → 透传
- `response`（带 id 命令回执）→ `success==false` 先发 upstream_error；无论成败都触发终止
- `message_start`/`message_end`/`turn_start`/`session_info_changed` → 忽略
- 保留对旧虚构格式的解析可以删除（测试同步改）

### 2. 修复事件路由 `PiRpcClient`（单活跃 prompt 模型）

因 PI scope=right_only + write_lock 串行，放弃"按 id 投 pending"：
- 新增 `self._active_prompt_id` / `self._active_queue`（在 `send_prompt` 写 packet 时于 lock 内设置）
- `_dispatch`：无 id 的流事件 → 投 `_active_queue`；`response` 的 id 匹配 active → 触发 `_PROMPT_DONE`（success=false 先投 upstream_error）
- 终止权威以 `response` ack 为准；`agent_end`/`turn_end` 作为内容流末事件透传，并作为兜底终止
- `_fanout_stream_closed` 改为面向 active prompt

### 3. 改写测试 `tests/test_agent_bridge_pi_rpc.py`

把所有 fixture 从虚构格式换成真实 pi 协议：`{"type":"message_update","assistantMessageEvent":{"type":"text_delta","delta":...}}`、`tool_execution_start/end`、不带 id 的流事件 + 带 id 的 `response` 终止。新增 `thinking_delta` 不进答案、`text_end` 兜底、`response success=false→upstream_error` 用例。

### 4. 真机探测确认协议细节（你的机器，沙箱不行）

复用 `_probe_rpc.mjs` 或临时 python 探测，发一个真实 prompt，dump 原始 stdout 行确认：
- `text_delta`/`thinking_delta` 字段与嵌套是否与参考一致
- `tool_execution_*` 字段名
- `agent_end`/`response` 的 usage 位置与时序（决定终止以谁为准）
- 关键：`aicodemirror/gpt-5.5` 在哪个 `PI_DEFAULT_THINKING` 档位会稳定回 `text_delta` 而非只回 reasoning（直接关系到是否还会"答非所问"）

### 5. 收尾

- 若存在 `_pi_probe.py` 临时脚本则清理（当前 glob 未发现，可能已删）
- 回归：`python -m unittest tests.test_agent_bridge_pi_rpc tests.test_agent_bridge tests.test_dashboard`，对齐既有绿基线
- 真机验证：右侧选 pi_agent 发问 → 首个 `message_delta` 是正式答案、内容切题；至少一次 tool_call+tool_result；agent_end 带 usage
