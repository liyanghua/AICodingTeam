## 冲突修复与规范对齐

当前 plan（G1-G6）与新规范存在 4 条明确冲突：

1. **PatchSet 形态**：plan 用单文件 `{target_path, edit_kind, anchor, replacements:[{old,new}]}`；规范要求 `patches:[]` 数组、`preserve_capabilities`、`verification`、`problem_source`
2. **edit_kind 枚举**：plan 自创 `replace_in_block`（anchor 定位 block + 逐条 replacements）；规范要求 `replace_block / append / create_file / replace_text`（无 `replace_in_block`）
3. **action 白名单**：plan 只追加 2 个；规范要求补 6 个（`patch_app / patch_artifact / diagnose_app_bug / verify_patch / rollback_patch / promote_patch_to_generation_rule`）
4. **Preview env 注入**：plan 不涉及；规范要求从仓库根 `.env` 提白名单 13 字段注入子进程，secret redaction 全链路

本轮修复 4 条冲突，让实现与规范形态对齐。5 条新增能力（见 § Phase 2 范围外）延后。

## 修复方案

### C1 — PatchSet 升级（单文件 → 批量事务）

**后端**：[`growth_dev/team/dashboard.py`](growth_dev/team/dashboard.py) `patch_app_generation_run` L1270 起

当前只接收单文件 `{target_path, edit_kind, anchor, new_content}`。规范要求：

```python
{
  "patches": [
    {
      "target_path": "generated_apps/<slug>/server.js",
      "edit_kind": "replace_text",
      "old_content": "const model = process.env.OPENAI_IMAGE_MODEL || \"gpt-image-1\";",
      "new_content": "const model = process.env.OPENROUTER_IMAGE_MODEL || process.env.OPENAI_IMAGE_MODEL || \"openai/gpt-5.4-image-2\";"
    }
  ],
  "preserve_capabilities": ["四阶段工作流", "产品图上传", "Prompt 下载"],
  "verification": ["node --check server.js", "node --check public/app.js"],
  "problem_source": "app_preview"
}
```

改动：

- 保留向后兼容：若 payload 含 `target_path` 单字段，自动包成 `patches=[{...}]`
- 新增 `patches:[]` 批量分支：先全部 dry-run 校验（路径、old_content 精确匹配），任一失败整体失败；全部通过后依次写 diff、覆写文件、更新 index
- dry_run 返回合并 diff，单次确认
- `preserve_capabilities` / `verification` / `problem_source` 作元数据写入 `app_patches/index.json`，不改执行逻辑（v1 验证仍走两阶段重启健康检查，不跑 action 声明的验证命令）

**前端**：[`dashboard/app_generation.js`](dashboard/app_generation.js) `handlePatchAppAction` L1389 起

- 同时支持单文件（`action.target_path`）和批量（`action.patches`）
- 批量 diff modal 展示合并 diff（已有 unified diff 渲染可复用）
- body 透传 `patches / preserve_capabilities / verification / problem_source`

### C2 — 废弃 replace_in_block，改用 replace_text

[`growth_dev/team/dashboard.py`](growth_dev/team/dashboard.py) `_apply_edit` L1245 起

移除 plan G5 的 `replace_in_block` 分支（anchor 定位 block + 逐条 replacements），新增：

```python
if edit_kind == "replace_text":
    if not old_content:
        raise ValueError("replace_text requires old_content")
    count = original.count(old_content)
    if count == 0:
        raise ValueError(f"old_content_not_found: {old_content[:80]}")
    if count > 1:
        raise ValueError(f"ambiguous_match: old_content appears {count} times")
    return original.replace(old_content, new_content)  # 一次精确替换
```

规则：

- `old_content` 必须在文件中精确出现 **1 次**

- 不需要 anchor（全文匹配）

### C4 — Preview env 注入（白名单 13 字段）

[`growth_dev/team/preview.py`](growth_dev/team/preview.py)（当前文件 grep 0 命中，需新增函数）

新增：

```python
_ENV_WHITELIST = {
    "IMAGE_PROVIDER",
    "OPENROUTER_API_KEY",
    "OPENROUTER_API_BASE_URL",
    "OPENROUTER_IMAGE_MODEL",
    "OPENROUTER_IMAGE_SIZE",
    "OPENROUTER_IMAGE_QUALITY",
    "OPENROUTER_IMAGE_OUTPUT_FORMAT",
    "IMAGE_REQUEST_TIMEOUT_MS",
    "OPENAI_API_KEY",
    "OPENAI_IMAGE_MODEL",
    "OPENAI_IMAGE_SIZE",
    "OPENAI_IMAGE_QUALITY",
    "OPENAI_IMAGE_OUTPUT_FORMAT",
}

def _inject_preview_env(repo_root: Path) -> dict[str, str]:
    """从仓库根 .env 提取白名单字段，返回注入子进程的 env dict。"""
    env_path = repo_root / ".env"
    if not env_path.exists():
        return {}
    injected = {}
    for line in env_path.read_text(encoding="utf-8").splitlines():
        stripped = line.strip()
        if not stripped or stripped.startswith("#") or "=" not in stripped:
            continue
        key, value = stripped.split("=", 1)
        key = key.strip()
        if key in _ENV_WHITELIST:
            injected[key] = value.strip().strip("'\"")
    return injected
```

`start_preview` 在 `subprocess.Popen` 前调用 `_inject_preview_env(repo_root)`，合并到 env。

## Phase 2 范围外（规范已定义但本轮不做）

5 条新增能力延后到独立 Phase：

1. **AppPreviewContext 派生**：`NodeContext.preview_status / preview_url / preview_health / provider_health / generated_app_capability_gaps` 作为独立子对象注入 prompt，不混在 files 里

5. **promote_patch_to_generation_rule 候选记录**：`promotion_candidates/*.json`，只记录不改模板

Phase 2 触发条件：AC-058 到 AC-064 需要验收时，或用户明确要求"Agent 驱动修复全流程"。

## 兼容性保障

- 单文件 `{target_path, edit_kind, old_content, new_content}` 向后兼容自动包成 `patches=[...]`
- `sync_env=true` 作 `inject_env=true` 别名
- 前端对 Phase 2 action（`diagnose_app_bug` / `verify_patch` / `rollback_patch` / `promote_patch_to_generation_rule`）stub 占位，不报错不阻塞
- 脚手架默认值（[`growth_dev/team/app_generation.py`](growth_dev/team/app_generation.py)）不动；用户实机规范已更新默认模型为 `openai/gpt-5.4-image-2`，让 patch_app 修复已生成应用即可

## 风险

1. **replace_text 全文匹配风险**：old_content 含空格、换行、注释差异时匹配失败 → 通过 dry-run 提前拦截，diff modal 让用户确认
2. **8KB 截断可能把锚点截掉**：server.js 当前 <4KB，远低于阈值；Phase 2 迁移到 app_patch_targets + read_artifact 彻底解决
3. **PI 可能不输出 patches 数组**：fallback 生成 draft action + 前端提示补充
4. **批量 PatchSet 任一失败**：事务保证，不写任何文件
5. **Preview env 注入白名单遗漏**：13 字段来自规范明确列举，可扩展
